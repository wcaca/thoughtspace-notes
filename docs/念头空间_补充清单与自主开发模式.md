| **商业化模式** | 需要提前决定：完全免费 / 一次性买断 / 核心功能免费+云同步订阅制。建议**核心记录/整理/观察功能永久免费本地使用，云同步与AI增强功能作为订阅点**，避免"觉察工具收费卡脖子"的负面观感 |
| **版本更新与发布节奏** | Tauri应用的自动更新机制需要提前设计（含签名校验，防止更新包被篡改） |
| **错误监控与遥测** | 需要在尊重隐私前提下做崩溃报告（如自托管 Sentry，且默认脱敏，绝不上传笔记正文），并让用户可完全关闭 |
| **可用性测试计划** | Phase 0原型完成后，必须找5-8个真实用户做"出声思考"测试，专门验证"拖拽是否顺手""几何结构切换是否直观"，这类空间交互的问题只靠自己开发是测不出来的 |
| **性能预算指标（具体数字）** | 需要写死量化目标而非"流畅"这种模糊描述：如 60fps @ 500节点可见、冷启动 < 2秒、内存占用 < 300MB @ 1000念头。这些数字要写进CI的性能回归测试阈值 |

## 二、自主工作模式设计：用开源模型复刻 Fable5 式的自主长程闭环

### 2.1 先拆解要复刻的核心特征

从公开信息看，这类"自主工作模式"的核心不是模型多聪明，而是**工程闭环的设计**：<cite index="10-1">按阶段规划、委派给子智能体、验证自己的产出</cite>，并且<cite index="6-1">能自主完成"跑测试→定位失败→改代码→再跑测试→确认通过"这一整套循环</cite>。这套模式的三个可复用要素是：

1. **Reason → Act → Verify 闭环**：不是一次性生成代码，而是"生成→自动验证→失败则自我修正"的迭代
2. **分阶段checkpoint而非逐行监督**：人类只在阶段边界审阅，不参与中间每一步
3. **子智能体委派**：大任务拆解后分给专职子agent，而不是一个agent从头做到尾

这三点和模型是否开源无关，是**agent工程架构**的问题，可以用开源模型复现，只是开源模型代码能力普遍弱于顶尖闭源模型，因此需要**更强的自动化护栏来补偿能力差距**——这是设计这套方案时最重要的前提。

### 2.2 技术选型：agent框架 + 开源模型

| 组件 | 推荐选项 | 说明 |
|---|---|---|
| 自主编码框架 | **OpenHands**（原OpenDevin）或 **Aider** | OpenHands提供完整的"沙箱执行环境+浏览器+终端"agent循环，适合长程自主任务；Aider更轻量，git集成好，适合"任务级"自主编码（一个任务一个commit），可作为OpenHands之外的补充工具用于快速小任务 |
| 多智能体编排（子智能体委派） | **MetaGPT** 或 **CrewAI** | 用于把Part3第十一节的Phase任务清单拆给"架构守卫agent""编码agent""测试agent""文档agent"等角色分工协作 |
| 开源代码模型（本地部署） | **Qwen2.5-Coder-32B-Instruct**（推荐首选）、**DeepSeek-Coder-V2**、**DeepSeek-V3**（若有能力自托管更大模型） | Qwen2.5-Coder-32B 在同尺寸开源模型中代码能力较强，32B级别单卡（如4090 48G或A100）可跑，性价比高；DeepSeek系列体量更大，能力更强但自托管门槛更高 |
| 模型托管方式 | 自托管：**vLLM**（推荐，吞吐高） + GPU服务器；或云端开源模型API：**Together.ai / Fireworks / DeepInfra / Groq** | 自托管保护代码隐私、长期成本可控；云端API起步快、无需运维，适合前期验证阶段先用云端API跑通流程，后期视用量决定是否转自托管 |
| 上下文/知识库 | **本地向量数据库（Chroma 或 Qdrant）** | 把 Part1-3 的所有规划文档拆分成chunk存入向量库，agent在生成代码前先RAG检索相关架构约束，避免"记性差"导致的架构漂移 |

### 2.3 自主循环架构设计

```
┌────────────────────────────────────────────────────────────┐
│  任务队列（Task Queue）                                        │
│  来源：Part3第十一节的分阶段工程任务清单（已经是可开票粒度）           │
└──────────────────────┬─────────────────────────────────────┘
                        │ 取出一个task
                        ▼
┌────────────────────────────────────────────────────────────┐
│  规划子agent（Planner）                                        │
│  - 从向量库RAG检索该task涉及的架构约束/接口定义                     │
│  - 拆解为具体的文件级修改计划                                     │
└──────────────────────┬─────────────────────────────────────┘
                        ▼
┌────────────────────────────────────────────────────────────┐
│  编码子agent（Coder，Aider/OpenHands驱动）                       │
│  - 按计划生成/修改代码                                          │
│  - 自动生成对应的单元测试（若任务清单未预先提供）                    │
└──────────────────────┬─────────────────────────────────────┘
                        ▼
┌────────────────────────────────────────────────────────────┐
│  验证子agent（Verifier）—— 这是补偿开源模型能力差距的关键环节         │
│  - 运行 Vitest / Playwright                                  │
│  - 运行 dependency-cruiser 架构规则检查                         │
│  - 运行 ESLint/TypeScript strict 检查                         │
│  失败 ──► 打包报错信息，退回 Coder子agent，最多重试N次（如5次）      │
│  通过 ──► 继续                                                │
└──────────────────────┬─────────────────────────────────────┘
                        ▼
┌────────────────────────────────────────────────────────────┐
│  提交子agent（Committer）                                      │
│  - 自动生成规范化commit message                                 │
│  - 提交到feature分支，推送，触发CI                                │
│  - 若连续N次验证失败仍未通过 → 标记为"需要人工介入"，暂停并通知        │
└──────────────────────┬─────────────────────────────────────┘
                        ▼
        [下一个task] ── 循环直到该Phase全部task完成
                        │
                        ▼
┌────────────────────────────────────────────────────────────┐
│  Phase边界人工checkpoint（唯一必须的人工参与点）                   │
│  - 审阅本阶段的功能演示（不是逐行看代码）                            │
│  - 决策是否调整下一阶段任务优先级                                  │
│  - 批准合并到主分支                                              │
└────────────────────────────────────────────────────────────┘
```

### 2.4 "用户最少参与"的具体落地方式

- **人工参与点严格限定在两类**：① Part1/3划分的5个Phase边界（约5次checkpoint，覆盖整个项目周期）；② 验证子agent连续多次失败后的异常升级
- **其余时间全部自动运行**：任务队列消耗、编码、测试、修复、commit全部由agent循环完成，可以挂在服务器上跑（类似"多日无人值守"），人类只需要隔一段时间查看进度看板
- **关键架构决策仍需人工拍板**（不适合完全自主）：是否新增一种几何结构、是否调整核心数据模型这类"改变系统设计"的决策，不应该让agent自主决定，只应该让agent在人工确认的方向内自主实现

### 2.5 用测试驱动来补偿开源模型的能力差距

这是整个方案能否成立的关键。开源模型在复杂代码生成上的可靠性目前普遍低于顶尖闭源模型，如果没有强力的自动化验证兜底，自主循环很容易产出"看起来对但实际有bug"的代码却无人发现。因此：

1. **测试先行**：Part3第十一节里每个task在进入agent循环前，先由人工（或agent在人工审阅下）为该模块编写测试用例大纲，作为agent自我验证的"锚点"，而不是让agent自己既写代码又写测试却没有独立标准
2. **几何引擎类模块要有"黄金测试集"**：比如 `TreeEngine.computeLayout()` 给定固定输入，断言输出的层级关系和坐标范围，这类可验证性强的模块最适合全自主循环；反之"结晶动画的视觉效果好不好看"这类主观判断，不适合完全交给自主循环，需要人工在Phase checkpoint时用眼睛看
3. **架构约束自动化**：`dependency-cruiser` 规则、ESLint规则必须在验证环节强制执行，任何违反直接判定失败退回，不给agent"绕过架构约束但能跑"的机会

### 2.6 风险与应对（开源模型自主开发专属风险）

| 风险 | 应对 |
|---|---|
| 开源模型代码能力波动，同样任务多次尝试结果不稳定 | 验证环节做严格，允许多次重试；对代码质量要求高的模块（如CRDT同步逻辑）优先人工编写或人工重点审阅，不完全依赖自主循环 |
| 长程任务上下文丢失、逐渐偏离架构约定（"架构漂移"） | RAG检索+每次生成前重新注入关键接口定义；dependency-cruiser作为最后一道硬性防线 |
| 自主修复陷入死循环（改了又错、错了又改） | 设置重试上限（如5次），超过自动标记人工介入，避免无限消耗算力 |
| 自托管GPU成本 或 云端开源模型API调用成本累积 | 前期用云端API（Together.ai等）低成本验证流程可行性，跑通后再评估是否值得投入自托管硬件；32B级别模型的推理成本远低于调用顶尖闭源模型API |
| 子agent之间协作产生冲突（如两个task同时修改同一文件） | 任务队列按文件/模块做依赖关系排序，避免并发冲突；必要时对同一模块的task强制串行执行 |
| 验证子agent本身也可能被开源模型"糊弄"（比如生成一个总是返回true的测试） | 关键模块的测试用例由人工编写或人工审阅后再固化，不允许agent同时拥有"写实现"和"写验证标准"的双重权力 |

### 2.7 具体启动步骤

1. 先在 Phase 0（Part3第十三节的最小验证原型）上试跑这套自主循环，用真实工程量小的任务验证流程本身是否顺畅，而不是一上来就让它跑整个MVP
2. 搭建：向量库（把Part1-3文档灌入）+ Aider/OpenHands + Qwen2.5-Coder-32B（先用云端API跑通，如Together.ai提供的Qwen2.5-Coder托管）
3. 把Part3第十一节Phase 0的task清单转成结构化任务（JSON/YAML），作为任务队列输入
4. 跑通"一个task从取出到自动commit"的完整单次循环，确认可行后再批量投喂剩余task
5. 全程记录每个task的重试次数、失败原因，作为后续判断"哪些模块必须转人工"的依据——大概率发现：几何算法这类逻辑清晰、可测试的模块自主循环成功率高；CRDT同步、动效细节这类模糊/强依赖主观判断的模块，自主循环成功率低，应主动收窄自主范围而不是硬推

## 三、这一层规划要回答的问题

自主开发模式的价值不在于"完全不需要人"，而在于**把人的精力从"写每一行代码"重新分配到"做架构决策和验收阶段性成果"**。所以设计这套流程时，真正要问的问题是：

> 我作为人类，应该在哪几个点上做判断，其余时间可以完全放手？

Part3已经把工程任务拆到可开票粒度，本篇的价值是把"谁来完成这些票"从默认的"你自己写"，转变为"agent自主完成，你只在5个Phase边界和异常升级时出现"。这个转变能不能真正生效，取决于第2.5节"测试驱动补偿"这一环是否扎实——这是比选哪个开源模型更重要的决定因素。
