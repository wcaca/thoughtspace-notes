# 方法 03 · 拓扑优先级表

> **TL;DR**:
> 1. **5 状态机** — 🟢 底座 / 🟡 焦点 / 🔵 沉淀 / ⚪ 废弃 / 🔴 孤儿
> 2. **每个文件 / spec / 原则** 都有状态
> 3. **修改规则按状态分** — 底座要 GEB 完整流程,焦点可快速迭代
> 4. **孤儿必决策** — 🔴 是"待你下决心"的标记

## §1 为什么需要拓扑表

**反模式**:项目大起来后,不知道:
- 哪些能改,哪些不能改
- 哪些是当前焦点,哪些是历史遗留
- 哪些有冲突,需要决策

**拓扑表** = **状态视图**——一眼看清全貌。

## §2 状态机

```
🟢 底座 ─→ 🟡 焦点 ─→ 🔵 沉淀 ─→ ⚪ 废弃
                      ↑
                      └──→ 🟢 底座 (重大变更)

🔵 沉淀 ─→ 🔴 孤儿 ─→ (任何状态) (决策完成)
```

### 2.1 各状态含义

| 状态 | 含义 | 修改规则 |
|---|---|---|
| 🟢 底座 | 不可修改 / 已锁定 / 几乎不变 | 必须走 GEB 完整流程 |
| 🟡 焦点 | 正在演进 / 活跃修改 | 可快速迭代,改动后更新本表 |
| 🔵 沉淀 | 完成 / 稳定 / 但仍可演进 | 修改需补丁或新 spec 替代 |
| ⚪ 废弃 | 不再使用 | **禁止修改**,仅供历史查询 |
| 🔴 孤儿 | 设计过时 / 与新规范冲突 / 状态不明 | **必须先决策**才能继续 |

### 2.2 状态转换触发

| 从 | 到 | 触发 |
|---|---|---|
| 🟢 → 🟡 | 重大变更 | 创建新 spec,在本表登记 |
| 🟡 → 🔵 | 验收完成 + 1 周无新需求 | spec §元信息加"状态:沉淀" |
| 🔵 → ⚪ | 被新设计替代且无引用 | spec §元信息加"状态:废弃" |
| 🔵 → 🔴 | 与新规范冲突 | 列入下个会话的议题 |
| ⚪ → 任何 | 重新启用 | 创建新 spec,**不修改旧 spec** |
| 🔴 → 任何 | 决策完成 | 在本表登记决策与新状态 |

## §3 表格结构

| 文件 / Spec / 原则 | 状态 | 创建 | 关联 | 备注 |
|---|---|---|---|---|
| `path/to/file` | 🟡 | 日期 | 其他模块 | 关键信息 |

**5 个标准表**:
- §2 规范状态表 (spec)
- §3 代码文件状态表 (src/...)
- §4 原则 / 教义 / 约束表
- §5 决策治理 (孤儿议题)
- §6 治理节奏

## §4 关键原则

### 4.1 修改前必查

> "每次大改动前,先查本表确认要修改的目标处于哪个状态、对应什么处理规则"

### 4.2 修改后必补

**诚实记录**: 我曾犯"承诺了改动后更新本表但没回头补"的错。

**修正**: 实施完每个 task,立即回头补本表状态。

### 4.3 孤儿必决策

🔴 孤儿 = **待用户决策**。不是"已知的麻烦",是**已知需要行动但还没行动的麻烦**。

每 1-2 周 **必须** 治理一次 §5 孤儿议题,转化为:
- ⚪ 废弃 (不解决)
- 🔵 沉淀 (接受了现状)
- 🟡 焦点 (准备解决)
