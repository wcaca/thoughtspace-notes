# 08 · 跨视角并行审查 (agent team review)

> **TL;DR (3 行)**  
> 1. **何时用** — sub-project 收尾前 / spec 定稿前 / PR merge 前 / 大重构前  
> 2. **派 N 个独立视角** — code / doc / governance / security / perf(挑 ≥3 个相关的)  
> 3. **诚实分级 P0/P1/P2** — P0 必修,P1 应修,P2 可留,**亮点也要记**

---

## §1 为什么要这个方法

### 1.1 单视角的盲区

一个 agent / 一个人的"完成感"无法靠内部反思打破:
- **代码审查员** 看不到文档可发现性差
- **文档审查员** 看不到边界 case 让 store 损坏  
- **治理审查员** 看不到代码 API 表面设计

每个视角都有自己的盲区,**正交视角**才能交叉发现。

### 1.2 "多视角并行"的 3 个强项

| # | 强项 | 价值 |
|---|---|---|
| **1** | 多视角并行 | 一个人的盲区被另一个视角打破 |
| **2** | 诚实分级 | P0/P1/P2 + 亮点,不让"完成感"掩盖问题 |
| **3** | 跨域独立 | 每个 agent 只看自己擅长的一块,没有局部最优互相迁就 |

### 1.3 反例:跳过审查的代价

thoughtspace-notes 的 SP-1 在我"完成"+ "全过" + "push 成功"后,**审查员找到 5 个 P0**:
- `applyObserveReorder` 函数不存在(文档/代码不一致)  
- pre-commit hook 从未安装(整套门禁物理上从未生效)  
- `process.cwd()` 假设仓库根(父目录跑 crash)  
- README 是 8 行空壳(新人无引导)  
- `methodology/"可独立 cp -r"`是伪命题(6/7 文件硬链接私域)

**这是 3 个自己绝不可能发现的问题**——独立视角是唯一解。

## §2 怎么派 agent team

### 2.1 视角选择

| 视角 | 适合什么时候派 |
|---|---|
| **代码审查** | 任何代码改动 ≥30 行 |
| **文档审查** | 文档变更 ≥3 文件 / 跨章节结构变化 |
| **治理审查** | 加新门禁 / 改状态机 / 改协议 |
| **安全审查** | 接口暴露 / 鉴权 / 输入校验 |
| **性能审查** | 算法变更 / 数据规模预期变化 |
| **可访问性** | UI 改动 / 键盘交互 / 颜色对比 |

**最少 3 个视角** — 比这次的 sub-project 是"代码 + 文档 + 治理"。

### 2.2 派 agent 的标准 prompt 模板

```
## 范围 (具体文件)
- 列出本次审查的具体路径

## 你的角色
**严格的角色审查员** — 不要客气,只关心 <特化维度>

## 工具
- Read / Grep / LS
- 跑相关命令验证

## 必须返回
1. P0 问题 (必修): <症状 / 位置 / 为什么 / 修复建议>
2. P1 问题 (应该修)
3. P2 问题 (锦上添花)
4. 亮点 (值得保留)

每条问题必须有:文件:行号 + 描述 + 建议修复
不要客气,直接列出。最终输出 Markdown 格式报告。
```

### 2.3 视角正交性检查

派完 agent 后**检查 prompt 是否真独立**:
- ✅ code 视角不读 docs/,doc 视角不读 src/
- ❌ 多个视角都查"测试是否通过"(浪费成本)

如果两个视角都查同一件事,**合并视角或拆细**。

## §3 怎么汇总报告

### 3.1 收到 N 份报告后

每份报告**独立读完**,再合并:

```
报告 A (代码) → 找出 P0 列表
报告 B (文档) → 找出 P0 列表
报告 C (治理) → 找出 P0 列表
↓
合并 + 去重(同一问题可能被多个视角发现)
↓
按修复优先级排序
↓
决定是真修(立即修) / 错峰修(下次会话) / 存档(已知但接受)
```

### 3.2 诚实分级的标准

| 级别 | 定义 | 处理 |
|---|---|---|
| **P0 (必修)** | 修不修都是事故(数据损坏 / 门禁不生效 / 跨平台 crash) | 当前会话必修 |
| **P1 (应修)** | 影响质量但不致命(不一致 / 缺字段 / 文档误导) | 当前会话修,但允许妥协 |
| **P2 (锦上添花)** | 性能 / 风格 / 风险低 | 可以下次会话或写 TODO |
| **亮点** | 好的设计决策 | **必须保留**(避免下次重构误删) |

### 3.3 决策矩阵

收到报告后**做决定 + 写下来** — 模板:

```markdown
## 修复决策

| # | 问题 | 级别 | 决定 | 理由 |
|---|---|---|---|---|
| 1 | applyObserveReorder 不存在 | P0 | 立即修 | SP-1 核心功能没接通 |
| 2 | _map 暴露 | P0 | 立即修 | 一行删除 |
| ... | ... | ... | ... | ... |
| 9 | Unicode \p{L} 支持 | P2 | 错峰修 | 当前 anchor 都是 ASCII,但下个 SP 可能踩坑 |
```

**重要**:决定**必须写下来** — 不写 = 没决定。

## §4 怎么用审查结果

### 4.1 立即修 P0

P0 必须**当前会话修完** — 不修不 push。

### 4.2 错峰修 P1

如果 P1 涉及大改 / 跨多文件,**单独 commit / 单独 PR**,本次会话先记录到 backlog。

### 4.3 存档审查报告

审查报告**复制到仓库** — 路径:
```
docs/audit/<YYYY-MM-DD>-<sp-or-feature>-quality-review.md
```

**为什么存档**:
- 6 个月后回看"上次发现什么"
- PR review 给 reviewer 看
- 案例教学(SP-N 收尾时可以参考 SP-1 的报告结构)

### 4.4 亮点要明示

写报告时**亮点也要记** — 否则下次重构可能误删好的设计。

---

## §5 何时**不**派 agent team

### 5.1 不要为小改动派

| 改动规模 | 建议 |
|---|---|
| < 30 行 / 单文件 | 自审 OK |
| 30-100 行 / 多文件 | 1 视角审查 |
| > 100 行 / 跨模块 | **3+ 视角并行审查** |
| sub-project 收尾 | **强制 ≥3 视角 + 存档报告** |

### 5.2 不要为已审查过的部分重派

如果上次审查已覆盖某区域,**只在变更区域重审**:
- 改了 src/core/*.js → 重审代码
- 改了 docs/methodology/* → 重审文档
- 全没改 → 跳过

### 5.3 报告不替代测试

agent team 找的是**结构性问题** — 测试覆盖率 / 性能压测 / 真实浏览器手感**仍要单独跑**。

## §6 与其他方法的关系

```
01 设计 → 02 spec → 03 拓扑 → 实现 → 06 架构守卫
                                              ↓ SP 收尾前
                            ┌────── 08 跨视角审查 ──────┐
                            ↓                          ↓
                       P0 修复 → 再跑门禁      审查报告存档
                            ↓
                    SP 升级为 🔵 沉淀
```

### 6.1 与 06 架构守卫的分工

| 方法 | 检查什么 | 怎么检查 |
|---|---|---|
| **06 架构守卫** | 代码 + 文档**结构合规** | 自动化脚本 (CI) |
| **08 跨视角审查** | **设计 / 完整性 / 可发现性** | 多 agent 并行读取 + 判断 |

**06 是机器能查的,08 是机器查不到的** — 二者必须并行,不可替代。

### 6.2 与 03 拓扑优先级表的衔接

`🔵 沉淀` 的语义要求:
- 测试 100% 过
- 门禁全过
- **≥1 份完整的 audit 报告(08 输出)**

`🔴 孤儿` 转 `⚪ 废弃` 不需要审查 — 没人维护的代码不可能有审查。

```markdown
## §3 P2 问题(锦上添花)

<按修复顺序排>

```
