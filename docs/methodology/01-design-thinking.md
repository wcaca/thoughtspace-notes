# 方法 01 · 设计思维

> **TL;DR**:
> 1. **分解优先** — 跨 4 个子系统的大型请求,先分解为 sub-project
> 2. **一次一问** — brainstorming 时不堆问题,每个 AskUserQuestion 1-4 选项
> 3. **设计 → 写 spec → 拿批准** — 不写代码先写设计文档
> 4. **整合已有材料** — 包含已有 spec + 用户拓展补足,**不抛弃任何方向**

## §1 为什么需要设计思维

**反模式**:用户问"我要做 X",你立刻开始写代码。
**结果**:做了 3 周,用户说"不,我想要 Y" → 3 周白做。

**设计思维**:在写任何代码前,**和用户对齐设计**。

## §2 流程

```
Step 1: 探索项目上下文
  - 看现有文件结构 / spec / plan / 最近 commit
  - 找到已有的相关模块

Step 2: 评估范围
  - 单个功能? → 直接进入 Step 4
  - 跨 3+ 子系统? → Step 3 分解

Step 3: 分解为 sub-project (如必要)
  - 每个 SP 独立 spec
  - 标注 SP 之间的关系和依赖

Step 4: 一次一问题地澄清意图
  - 不要堆 5 个问题
  - 用 AskUserQuestion 提供 2-4 选项 + "Other"
  - 在每题后等用户回答

Step 5: 提出 2-3 个方案 + 推荐
  - 列出每个方案的 trade-off
  - 说明你推荐哪个 + 为什么

Step 6: 写出 spec 文档
  - 路径: docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
  - 含: 核心洞察 / 数据模型 / UI 架构 / 验收标准 / 风险
  - 含: "用户原话对照表"(防信息丢失)

Step 7: 用户批准 spec
  - 用户说"批准" → 进入 writing-plans skill
  - 用户说"改 X 节" → 编辑后回到 Step 7

Step 8: 写作 plans + 开始实施
```

## §3 关键原则

### 3.1 范围分解信号

> "if the request describes multiple independent subsystems, flag this immediately... help the user decompose into sub-projects"

**示例**:用户说"看板可分层 + 块嵌套 + 3D 反映" → **3 个独立子系统** → 拆 SP-1/SP-2/SP-3。

### 3.2 一次一问的反模式

❌ 错: 一次问 5 个问题
```
1. 你想要几个层?
2. 块的内容是什么?
3. 排序按什么?
4. 嵌套深度?
5. UI 在哪?
```

✅ 对: 一次一个问题,基于回答追问
```
Q1: "分层方式有 4 种,你倾向哪个?"
A1: "倾向第一种,但其他方式也很有价值"
Q2: "其他方式的价值你具体指什么?"
A2: "方形空间 vs 圆形空间认知不同"
Q3: "这是几何学原理,可以帮我把它形式化吗?"
...
```

### 3.3 整合不抛弃

用户说"我倾向 X,但其他方式也很有价值" → **不要立刻抛弃其他方式**,**整合**:

> "倾向第一个的原因是我们与屏幕界面的交互视角影响很大,这个你理解吗"

→ 把"屏幕 2D 投影"作为**几何学原理**形式化,而不是只选第一个方案。
