### 1.2 首次通读顺序（给你自己看的）

**不建议按Part1到Part9的生成顺序死板通读**，因为生成顺序是"逐步深挖"的对话顺序，不是"由浅入深"的最佳阅读顺序。建议：

1. **先读 Part1** 全文——建立产品的核心心智模型（笔记→念头→结构→行动的转化链条），这是理解后面一切的地基
2. **再读 Part2 第一、二节**（不用逐个几何结构细节都看）——理解"形状即语义"这个设计哲学即可，12种结构的具体规格留到真正要实现某一种时再回来查
3. **跳到 Part8 第三节"合理留白清单"**——提前知道哪些数字是"暂定值，需要实践中校准"，避免读Part3/Part4时把里面的具体数字当成绝对真理
4. **读 Part3 全文**——这是真正要动手前必须完整吃透的一份，尤其第二节分层架构和第十三节"最小可验证起点"
5. **读 Part4 全文**——如果你打算真的用自主开发模式，这份是流程骨架
6. **Part5/6/7 按需精读**——这三份是"让Part4跑得更好"的增强机制，如果你打算先手动开发Phase 0再决定要不要上自主循环，这三份可以先泛读留印象，等真正要搭建agent系统时再回来逐节细读

### 1.4 给"AI agent自己"的喂入顺序（区别于给人看）

如果按Part4-6设计真正搭建自主开发系统，喂给agent知识库的顺序和人类阅读顺序不同：**agent不需要"循序渐进理解"，需要的是"每次都能精确检索到当下task相关的内容"**。因此：

- 项目宪法文档（Part6 1.5节草稿）→ 无条件全文注入所有agent会话，不走检索
- 其余Part1-8全部按Part5的方案切分chunk存入向量库，供按需检索，**不需要也不应该要求agent一次性"读完"全部八份文档**——这是人类阅读模式和agent检索模式的本质区别，不要用人的阅读习惯去要求agent

### 1.5 哪些是"一次性定稿"，哪些是"活文档"

| 文档 | 性质 | 更新方式 |
|---|---|---|
| Part1 产品功能规划 | 相对稳定，除非产品方向大改 | 低频，需要时手动修订 |
| Part2 几何结构系统 | 相对稳定 | 低频，新增几何结构时才需要追加 |
| Part3 技术架构 | **活文档** | 随技术选型调整持续更新，建议和代码一起进git做版本管理 |
| Part4-7 工程流程类 | **活文档，且应该主要靠实践反馈更新** | Part7提到的"知识蒸馏飞轮"机制本身就要求这几份文档随Phase推进不断补充案例和调整参数 |
| Part8 颗粒度补全 | **专门用来被后续实践"打脸"和修正的文档** | 每次实践中确定了一个"留白清单"里的数字，就应该回来更新这份文档，把它从"待定"改成"已确定：xxx，确定依据：xxx" |
| 本篇（Part9） | 活文档 | 每次实际发生一次新的意外情况并总结出应对方法后，应追加进第三节 |

## 二、执行指导：怎么真正开始动手

### 2.1 立即可做的第一步（不要跳过）

严格按Part3第十三节 + Part6第1.2节的顺序：

1. **先写Part6 1.5节的项目宪法文档实体文件**，存入代码仓库根目录（如 `PROJECT_CONSTITUTION.md`），这是所有后续工作的锚点
2. **搭建 Part3 第十三节的最小原型**（Yjs+PixiJS+IndexedDB+cluster布局的可拖拽画布），这一步**不使用自主开发模式**，建议人工亲自实现或用常规AI辅助编码（如Claude Code/Cursor）完成，因为这是验证核心交互手感的阶段，需要人对"顺不顺手"做直接判断，不适合一开始就上自主循环
3. 原型验证通过后，**再决定是否要搭建Part4-7的完整自主开发系统**——如果项目规模不大、你愿意亲自写大部分代码，Part4-7整套机制可以只取用其中"知识库辅助检索"这一部分（用向量库存文档+代码，日常编码时按需检索），不必上完整的自主agent循环，这是一个可以按需裁剪的方案，不是全有全无

### 2.2 分阶段执行检查清单

每个Phase开始前，确认：
- [ ] Part1/Part3对应Phase的任务清单已转成具体的task列表（Issue/Ticket）
- [ ] Part8"合理留白清单"里本阶段会涉及到的参数，是否已经从上一阶段的真实数据中得到校准值
- [ ] 项目宪法文档的"当前所处阶段"和"当前待解决问题"两节是否已更新

每个Phase结束前，确认：
- [ ] Part7第2.1节的"知识沉淀"整理是否已完成
- [ ] Part8的合理留白清单是否有新增可确定的数字，需要回填
- [ ] 本篇第三节的异常处理记录是否有新案例需要补充

## 三、编程中意外与混乱的应对手册

### 3.1 问题严重程度分级

| 级别 | 定义 | 响应方式 |
|---|---|---|
| **L1 - 常规失败** | 单个task验证不通过 | Part4标准重试循环自动处理，无需人工介入 |
| **L2 - 局部异常** | 重试上限用尽（5次）仍失败 | 自动标记人工介入，暂停该task，其余task继续 |
| **L3 - 系统性问题** | 同类失败在多个task中重复出现 / 架构约束被多次违反 | 暂停整个Phase的自动化循环，人工排查根因 |
| **L4 - 严重事故** | 数据损坏 / 生产环境影响（若已上线）/ 发现方向性设计错误 | 立即触发"紧急制动"（3.4节），全面停止自主循环，人工主导恢复 |

### 3.2 具体场景应对指南

**场景A：agent陷入"改了又错、错了又改"的死循环**
- 诊断：查看Part4/6记录的重试历史，是否每次修改都是同一个思路的变体（说明模型理解错了根本问题，而非代码细节错误）
- 应对：不要继续加重试次数。触发Part7的"孵化模式"——把task标记为pending，让agent先处理其他task，间隔一段时间/上下文后再回来重新审视，往往能跳出局部思路陷阱；若仍无效，直接转人工

**场景B：发现agent生成的代码违反了架构约束（如core-domain误依赖了渲染库）**
- 诊断：这类问题应该被 `dependency-cruiser` 在验证环节自动拦截，如果流入了主分支，说明验证流水线本身有漏洞
- 应对：① 立即revert该次合并 ② 检查CI配置为什么没拦截住（配置遗漏还是规则本身有盲区）③ 修补验证流水线漏洞后才能重新放行同类task，**不能只修复这一次的代码而不修补检测机制**，否则同类问题会再次流入

**场景C：知识库检索到过时/错误信息，导致连锁错误扩散到多个模块**
- 诊断：查Insight Ledger（Part7）和KB元数据的`last_updated`字段，定位错误信息的源头chunk
- 应对：① 立即将该chunk标记`deprecated`并从检索结果中排除 ② 用Part7 2.1节的"传播扫描"机制反向排查——既然这个错误信息被多个模块引用过，需要检查这些模块是否也需要连带修正 ③ 补一条ADR记录这次事故的根因，写入KB-3作为"失败尝试记录"的一种特殊类型

**场景D：多个子agent同时修改同一文件产生冲突**
- 诊断：查Part4任务队列的调度日志，确认是否遗漏了Part4 2.6节"同一模块task强制串行"的调度规则
- 应对：立即修正任务调度器的依赖关系配置；冲突产生的代码用标准git merge conflict流程人工解决（不建议让agent自主处理merge conflict，这类冲突往往涉及对两边意图的判断，风险较高）

**场景E：算力/API成本短期内异常飙升**
- 诊断：检查是否某个task陷入了未被及时发现的重试循环，或Part7的"顿悟增强机制"（Tree-of-Thought、多模型交叉校验）被过度触发到了不该用的简单task上
- 应对：立即检查Part7第五节的"决策表"分级策略是否在实际调度中被正确执行（很可能是Planner子agent的判断逻辑有bug，把简单task误判成了需要高强度处理的复杂task）；设置硬性的每日成本上限告警，触发后自动暂停循环等待人工确认

**场景F：Phase checkpoint人工审阅时，发现整个阶段的方向性理解都是错的**
- 这是最严重也最容易让人手足无措的情况，处理原则：
  1. **不要试图"修补"已经产出的代码去凑合方向**，方向错了修补只会让技术债更深
  2. 先冻结当前分支（不删除，保留作为"这条路走不通"的历史记录）
  3. 回到项目宪法文档，重新审视是宪法本身表述不清晰导致agent理解偏差，还是agent检索/推理环节出了问题——**这个判断决定了下一步是先修文档还是先修流程**
  4. 更新项目宪法文档使其更明确后，作为一条重要ADR记录这次教训，再重新开始该Phase的task
  5. 这类事故本身要作为高价值案例，详细写入本篇第3.5节的"事故档案"，因为这是最能暴露"顿悟机制/知识库设计哪里还不够"的真实数据

### 3.3 决策升级流程图

```
问题发生
   │
   ▼
是否是L1（验证失败）？──是──► 自动重试循环处理，无需人工
   │否
   ▼
是否是L2（重试上限用尽）？──是──► 标记介入，暂停该task，人工在下次checkpoint统一处理（不打断其他task）
   │否
   ▼
是否波及多个task/模块（L3）？──是──► 暂停该Phase自动化循环，人工立即排查根因，修补机制后才恢复
   │否
   ▼
是否涉及数据损坏/方向性错误（L4）？──是──► 立即"紧急制动"（3.4节），全面停止，人工主导
```

### 3.4 紧急制动机制（Kill Switch）设计

- **触发方式**：预设一个明确的暂停指令（如向任务调度器写入一个`HALT`标志文件，或调用一个专门的暂停API），任何人工（不需要是原本负责该项目的人）都应该能一键触发
- **触发后的行为**：
  1. 当前正在执行的task允许完成手头这一步（不强行kill进程导致数据写入到一半），但**不再从任务队列取新task**
  2. 自动生成一份"制动时状态快照"：当前所有task的状态、最近的Insight Ledger条目、最近的失败记录，供人工快速了解"停在哪儿了"
  3. 明确通知（而非静默停止），确保人工确实知道系统已暂停，避免误以为在正常运行
- **恢复方式**：必须由人工显式确认"问题已排查/已修复"后手动恢复，不允许自动超时恢复

## 四、这一层规划要回答的问题

前八份文档教会系统"该怎么正常运转"，这一份的价值是回答：**当系统没有按预期运转时，人应该在多快的时间内、以什么方式介入，介入到什么程度**。一个自主开发系统的可信度，不取决于它平时跑得多顺，而取决于**出问题时能不能被及时发现、清晰定位、安全止损**——3.4节的紧急制动机制和3.2节的分场景应对指南，就是为了保证"用户最少参与"不会滑向"用户完全失控"。

真正决定这套系统能不能长期稳定运行的，往往不是Part1-8设计得多精巧，而是第3.5节那份"事故档案"能不能被认真维护——每一次真实的混乱，处理完之后有没有变成让系统下一次更不容易犯同样错误的养分。
