## §6 新增原话 SOP

### 6.1 何时新增

| 触发 | 动作 |
|---|---|
| 用户在对话中产生新的产品哲学 / 决策 / 设计原则 | ✅ 新增原话条目 |
| AI 提议被用户认可并写入 spec | ✅ 新增原话条目（用户原话 + AI 提议合并记） |
| 复盘发现实装偏离用户原始意图 | ✅ 登记偏差，不修改原话 |
| spec §1 段落被新原话扩充 | ✅ 双向更新（spec §1 + 本档案） |

### 6.2 新增格式

```markdown
### Q-XXX · 一句话摘要

> **原话全文**："完整引用，不改一字"

**主题**：分类标签（产品灵魂/核心实体/空间哲学/物理属性/形状哲学/排序与多轴/工程哲学/拆解与协作）
**关联 spec/plan**：file:///...
**关联 spec §X**：file:///...#LX-LY
**会话日期**：YYYY-MM-DD
**实装状态**：✅ 完整对齐 / ⚠️ 部分对齐 / ❌ 实装偏离
**偏差记录**：如有，附 audit 报告链接
```

### 6.3 双向链接规则

1. **每条原话** → 必须挂在至少一个 spec/plan 的 §X 章节
2. **每个 spec §1** → 必须引用至少一条本档案的 Q-ID
3. **每个 audit 偏差** → 必须引用至少一条本档案的 Q-ID
4. **新增原话时** → 必须同时更新本档案 + 关联 spec/plan（不允许只更新一处）

### 6.4 维护节奏

| 触发 | 动作 |
|---|---|
| 新增用户原话 | 立即登记本档案 |
| 新增 spec | 在 spec §1 增加"用户原话"段，引用本档案 Q-ID |
| 实装状态变化 | 更新本档案 §5 实装状态对照表 |
| Round 反馈（如 Round 7） | 启动新一轮偏差检查，更新 §5.2 |
