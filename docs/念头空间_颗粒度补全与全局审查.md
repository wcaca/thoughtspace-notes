## 一、关键缺口的具体补全

### 1.1 念头温度衰减——从"有这个属性"到具体公式

Part1 1.3节只说"久置未碰逐渐冷却"，未给出可执行公式。补全：

```
temperature(t) = temperature_last_interaction × exp(-λ × Δt_days) + Σ(mass_boost_events)

其中：
- λ（衰减常数）默认 0.05（约14天温度降至初始值的一半，可在设置中调整，范围建议 0.02-0.15）
- Δt_days = 距上次该念头被交互（编辑/连接/查看详情）的天数
- mass_boost_events：每次新增一条指向该念头的Edge，temperature += 0.15；每次用户手动编辑该念头，temperature 直接重置为 1.0
- temperature 始终 clamp 在 [0, 1] 区间

状态映射：
- temperature ≥ 0.6 → 视觉呈现"活跃发光"
- 0.2 ≤ temperature < 0.6 → 常规状态
- temperature < 0.2 → 视觉呈现"冷却"（颜色变灰、体积缩小至原尺寸的70%）
- temperature < 0.05 且连续 ≥ 30 天无交互 → 触发"是否归档"的提示（非自动执行，需用户确认，呼应"提示而非执行"原则）
```

### 1.2 结晶判定——从"结构强度达到阈值"到具体计算方法

Part2/Part4多处提到"cohesion_score达到阈值触发结晶"，未给出计算方法。补全：

```
cohesion_score = 0.4 × 边密度得分 + 0.3 × 连接度均衡得分 + 0.3 × 用户确认得分

- 边密度得分 = min(1, 实际边数 / (节点数 × 1.5))
  （经验值：一组念头平均每个节点1.5条边以上视为"关系已经想清楚"）
- 连接度均衡得分 = 1 - 归一化后的连接度方差
  （避免"一个节点连接所有其他节点、其余互不相连"这种伪高密度的星型结构被误判为高内聚）
- 用户确认得分 = 用户是否点击过"这个结构我想清楚了" → 是记1，否记0

触发规则：
- cohesion_score ≥ 0.7 → 在画布边缘出现结晶候选提示（气泡，不打断操作）
- 用户主动点击确认 → 才真正执行结晶动画，写入 crystallizedForm 字段
- 未达到0.7但用户仍主动要求结晶 → 允许（不强制阻拦，只是不会主动推荐）
```

### 1.3 结构自动识别的完整量化规则（补全Part2表格里未给数字的行）

Part2第五节表格部分行已有数字，此处补全遗漏的具体计算方式：

- **矩阵坐标标准化**：所有几何结构的坐标类字段（`axisXValue`、`axisYValue`、`axisValue`、`radiusValue`）统一存储为 **0-1 的归一化浮点数**，UI渲染层再映射为用户可读的展示值（如映射为"低/较低/中/较高/高"五档标签，或映射为用户自定义的0-10整数刻度）。**统一内部存储标准，是为了让不同geometryType之间做数值层面的换算和比较时不需要反复做单位转换**
- **曼陀罗默认扇区模板**：默认提供"生活轮盘"8扇区模板（事业、财务、健康、人际关系、情绪、个人成长、休闲娱乐、居住环境），用户可自定义增减扇区（建议上限12个，超过则视觉拥挤，UI层做硬性限制提示）
- **网状结构高密度判定**：边数/节点数比值 ≥ 1.5 时判定为"网"（Part2已给出此数字，此处确认无需修改）

### 1.5 项目宪法文档——从"应该有这么一份文档"到实际草稿

## 四、总结：颗粒度审查的核心结论

七份文档在"体系设计"和"机制逻辑"层面已经完整（这部分本来就应该在规划阶段做到位），本篇补全的主要是三类此前遗漏的具体化内容：**可执行的计算公式**（温度衰减、结晶判定）、**可直接使用的实际文本样例**（项目宪法草稿、agent prompt模板）、**此前完全遗漏的合规检查项**（License兼容性）。

同时明确了一份"合理留白清单"——不是所有内容都应该在规划阶段量化，**色彩美术、行为参数校准、成功率阈值这类必须依赖真实数据或专业设计判断的事项，现在写死具体数字反而是伪精确，会在实践中造成误导**。真正的"最小颗粒度"，不是把一切都变成数字，而是**清楚区分"现在就能且应该量化的部分"和"必须留到有真实数据后才能量化的部分"，并且对后者明确写清楚"将在什么阶段、依据什么来确定"**——这才是规划文档在颗粒度上真正完整的标志。
