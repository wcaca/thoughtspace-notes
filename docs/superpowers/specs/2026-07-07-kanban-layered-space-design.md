# 看板分层与双模式空间 — 设计 Spec (SP-1)

> 把"看板视图"从一个 2D 表格升级为**认知空间的双模界面**：
> - **背景分区模式**：表达意识的 3D 空间结构（层 × 区）
> - **块模式**：表达念头/记忆的内容组织（块、嵌套、富内容）
>
> 这一 spec 只覆盖 **SP-1**（背景分区 + 双模式架构 + 几何学原理）。
> SP-2（块的内容数据模型）和 SP-3（3D 空间映射联合验证）单独 spec。

| 项目 | 值 |
|---|---|
| 创建日期 | 2026-07-07 |
| 状态 | **设计阶段 — 未批准前不实现** |
| 优先级 | P1 体验进化（follow-up 到拓扑意识空间 + 形状哲学） |
| 周期 | 5-7 天 |
| 前置依赖 | 拓扑意识空间 spec（已就位）+ 形状哲学 spec（已就位）+ zone.js（已有） |
| 后续 | SP-2 内容块数据模型，SP-3 3D 空间联合验证 |

## §0 项目分解声明

**原始请求跨 4 个独立子系统**：

1. 看板可自定义分层（空间几何）
2. 每层内自定义分块（空间几何 + UI）
3. 块是笔记模块（数据模型）
4. 块可嵌套 + 富内容（数据模型）
5. 排序需保留信念信息（哲学约束）
6. 3D 空间反映（视觉）

> **用户原话**："拆为 3 个 sub-project (推荐)。顺便做好我的记录、你的高质量信息的记录，防止编程进行中因为大模型能力和编程系统因素丢失关键信息"

**分解**：

| Sub-project | spec | 范围 |

| **SP-1**（本文档） | 看板分层与双模式空间 | 背景分区模式（层 × 区）+ 块模式 + 排序哲学 + 召唤多轴 + 几何学原理应用 |
| **SP-2**（后续 spec） | 内容块数据模型 | 树状笔记块 + 文字图片 + 与 Thought 集成 |
| **SP-3**（后续 spec） | 3D 空间联合验证 | 把 SP-1/SP-2 成果映射到 3D 空间 + 几何学原理（拓扑 / 边界 / 对称性）反复迭代 |

## §1 核心洞察（从用户原话提取）

### 1.1 用户的几何学直觉（关键！）

> **用户原话**："倾向于第一个的原因是我们与屏幕界面的交互视角影响很大"

用户的"第一个倾向"基于一个**视觉-认知几何学原理**：

> **信息呈现的"自然维度"应与人类视觉-认知通道对齐**——
> Y 轴（显意识→潜意识）天然映射到屏幕的上下方向（与重力一致）。
> 这是认知负担最小的映射。

这个原理不在既有 spec 里，是用户在 SP-1 brainstorming 中**直觉发现的**——它是**用户-屏幕交互的几何学**，需要被正式纳入设计。

### 1.2 用户的"不压制信念排序"原则（最高优先级）

> **用户原话**：
> "如果有一种固定的排序方式进行规定性，可能会让这种信念信息被压制"
> "嵌套问题也是刚刚说的念头/记忆两种模式的时候，应该隐含了这些信息"
> "我的直觉感觉是信息充足的，你可以推理出最佳合理科学的方式"

**原则 SP-1.P0（最高优先级，不可妥协）**：
- 排序与嵌套**不应由系统强制**
- 排序与嵌套**应从用户行为中学习**或**保留多种维度同时存在**
- 系统不预设"正确排序"——排序是用户表达信念的载体
- 多轴并列 + 拖动记录 = "信念轨迹"

### 1.3 用户的"高级 UI 设计技巧"约束

> **用户原话**：
> "多轴有时候会被当做用户理清自己思路，一种思考角度的使用方法"
> "但如果默认出现，又会打扰用户"
> "这里需要一种高级的 UI 设计技巧"

**原则 SP-1.P1**：
- 多轴能力**存在但默认隐藏**
- **用户召唤才出现**
- 召唤手势本身应**优雅、不打扰**

### 1.4 一句话定位

**看板视图升格为"认知空间双模界面"——背景分区模式展示空间结构，块模式展示内容组织；两者通过顶部 tab 切换；排序轴不强制出现，遵循"信念轨迹"原则。**

## §2 核心原则（设计阶段的产物，需经用户批准）

### 2.1 数据模型分层（不破坏既有）
> 双模式 + 6 层 + 多轴 是**视图决策**，不是**数据决策**。

- `Thought`（已有）保持不变：`id / text / body / x / y / z / temperature / labels / order / ...`
- 新增 `LayerStore`（层管理） + `CanvasMode`（模式状态机）
- 新增 `SortAxis`（排序轴） + `SortHistory`（信念轨迹）
- 任何视觉切换通过 CSS / Three.js mesh 几何 / transition 实现，绝不"重新创建"念头

### 2.2 几何学原理应用（用户直觉的形式化）

| 几何维度 | 原理 | 应用 |
|---|---|---|
| **拓扑（连续 vs 离散）** | 离散 = 可数可命名可指向；连续 = 比例密度节奏 | 层 = 离散（背景分区）；轴 = 连续（排序比例） |
| **边界 vs 主体** | 显意识 = 主体焦点；潜意识 = 边缘集合 | 显意识层靠近中心（明亮）；潜意识层靠近边缘（暗淡） |
| **对称性（中心 vs 动中心）** | 中心是注意力的汇集点；从边缘到中心是注意力汇聚过程 | 召唤多轴默认折叠在角落（边缘），召唤时汇聚到中心 |
| **屏幕 2D 投影** | Y 轴天然映射到屏幕上下；X 轴水平扩散 | 看板列方向 = Y 轴层（垂直）；块排布 = X 轴概念扩散（水平） |

### 2.3 不压制信念排序（SP-1.P0）

排序 = 用户的信念载体。系统提供：
- **默认按时间**（最弱信号，最不压制）
- **拖动记录**（用户主动表达）
- **多轴并列**（不同思考角度并存）

排序历史 = **信念轨迹**——可查看、不可改。

### 2.4 高级 UI 召唤（SP-1.P1）

多轴能力存在但默认隐藏。召唤交互：
- **顶部 tab 双模式切换**：背景模式 | 块模式
- **排序轴召唤**：长按顶部空白处 → 排序轴从顶部滑入（300ms 缓动）
- **召唤手势反馈**：召唤时轴从边缘向中心汇聚（对应几何学"对称性"原理）

## §3 数据模型

### 3.1 LayerStore（层管理）

```javascript
// src/core/layer-store.js — 新增
{
  layerById: Map<layerId, Layer>
}
Layer {
  id: string,
  name: string,          // 用户自定义："显意识层-1"
  color: string,        // 颜色锚定
  order: number,        // 0..N；0 = 最顶（显意识）
  kind: 'conscious' | 'subconscious',  // 软分类（不强制）
  visibility: 'visible' | 'collapsed',  // 折叠不影响数据
  createdAt: number
}

createLayerStore() → {
  add({name, color, kind, order?}),
  update(id, patch),
  remove(id),
  list(),
  reorder(newOrder),
  toJSON() / fromJSON()
}
```

### 3.2 CanvasMode（模式状态机）

```javascript
// src/render/canvas-mode.js — 新增
canvasMode = 'background' | 'block'
// 切换通过顶部 tab；
// mode 决定 observe-views 内的不同渲染路径
```

### 3.3 SortAxis & SortHistory

```javascript
// src/core/sort-axis.js — 新增
SortAxis {
  key: 'time' | 'heat' | 'volume' | 'manual',
  label: string,
  direction: 'asc' | 'desc'
}

SortHistory {
  axes: SortAxis[],         // 用户激活过的轴
  manualOrder: thoughtId[], // 用户手动排序的最终状态
  evolution: { timestamp, axes, manualOrder }[]  // 历史轨迹
}
```

**重要**：手动排序记录后，**不再覆盖**——这是"信念轨迹"，可看不可改。

## §4 UI 架构

### 4.1 看板顶部结构（双模式 tab）

```
┌─────────────────────────────────────────────────────┐
│  [背景模式] [块模式]                  [排序轴召唤]     │  ← 顶部 tab
├─────────────────────────────────────────────────────┤
│  ─ 显意识层 1 ─  [块 1.1] [块 1.2] [块 1.3]          │  ← 背景模式
│  ─ 显意识层 2 ─  [块 2.1] [块 2.2]                    │
│  ─ 显意识层 3 ─  [块 3.1]                             │
│  ─ 潜意识层 1 ─  [块 4.1] [块 4.2]                    │
│  ─ 潜意识层 2 ─  [块 5.1]                             │
│  ─ 潜意识层 3 ─  [块 6.1] [块 6.2] [块 6.3]          │
└─────────────────────────────────────────────────────┘
```

块模式下：

```
┌─────────────────────────────────────────────────────┐
│  [背景模式] [块模式]                  [排序轴召唤]     │
├─────────────────────────────────────────────────────┤
│  念头卡片（按选定轴排序，可拖动重排）                   │
│  [块 A]  [块 B]  [块 C]  [块 D]  ...                 │
│  [块 E]  [块 F]  [块 G]                              │
│  按 [时间 ↓] 排序 · 拖动调整后保存到「信念轨迹」       │
└─────────────────────────────────────────────────────┘
```

### 4.2 排序轴召唤交互

默认：顶部右下角**只有一个小的"召唤"图标**（类似"⤓"或自定义符号）。

长按（>600ms）顶部空白处 → 排序轴条从顶部滑入（300ms 缓动）。

排序轴条展示当前激活的轴 + 切换按钮。再次长按或点击 ✕ 收起。

### 4.3 模式切换动效

- 背景 → 块：层分隔线淡出，块平铺展开（400ms 缓动）
- 块 → 背景：块缩回对应层（层分隔线从空淡入）（400ms 缓动）

## §5 几何学原理的实现约束

| 几何维度 | 实现约束 | 验收 |
|---|---|---|
| **Y 轴 → 屏幕上下** | 层 0 在屏幕顶部；层 N 在屏幕底部；与 CSS flex column 一致 | 单元测试：layer 0..N 渲染时 `getBoundingClientRect().top` 单调递增 |
| **X 轴 → 水平扩散** | 块在层内横向排布；列数自适应视口宽度 | 单元测试：块 row 内 `getBoundingClientRect().left` 单调 |
| **召唤轴从边缘汇聚中心** | 轴条从屏幕顶部滑入，伴随透明度 0→1 与 scaleY 0→1 | 视觉测试（手工） |
| **背景模式 vs 块模式的对称性** | 切换的动效互为镜像（时间一致、缓动一致） | 单元测试：transition 时长与缓动函数匹配 |

## §8 风险与未决项

### 8.1 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 块模式可能破坏 T11 已修复的 observe-views 切换逻辑 | 高 | Task T3-T6 严格保持现有 onSwitch 路径不变；新逻辑只在 mode='background' / mode='block' 分支 |
| 多轴 UI 默认隐藏但用户找不到 | 中 | 提供 onboarding tooltip 一次（不强制） |
| 信念轨迹数据爆炸 | 低 | 限速：每分钟最多记录一次 |

### 8.2 未决项（依赖用户后续输入）

1. 排序轴**具体哪些维度**？默认：时间、热度、体积、手动。是否需要更多（如创建位置、最后交互时间、标签密度）？
2. 背景模式的"块"是抽象容器还是直接显示念头卡片？（SP-2 决定）
3. 召唤手势：**长按**还是其他（如滑下、二指轻点）？
4. 是否需要"信念轨迹可视化"功能（按时间回放手动排序变化）？这可能是 SP-3 的子任务。
