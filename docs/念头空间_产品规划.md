# 念头空间（暂命名）产品功能梳理与实现规划

> 一款"笔记 → 念头 → 结构 → 觉察"一体化的思维游乐场
> 定位：不是笔记软件+冥想App+Todo工具的拼接，而是同一个"念头空间"在不同粒度、不同视角下的呈现。

---

## 一、先把核心概念模型定死

在写功能列表之前，必须先回答几个决定成败的概念问题，否则功能会越做越散。

### 1.1 核心实体与转化链条

```
原始笔记 (Note)
   │  提取/拆解
   ▼
念头 (Thought) —— 空间中的最小可操作单元（类似"卡片/粒子"）
   │  拖拽、连接、聚合
   ▼
结构 (Structure) —— 念头之间的关系图（树/网/星团/时间线等）
   │  持续演化、被观察
   ▼
成型结构 (Crystallized Structure) —— 高度组织化后，可视化为"作品/器官/星系"等固定形态
   │  可以被"执行化"
   ▼
行动 (Action/Todo) —— 从结构中提炼出的可执行项
```

**关键设计原则：念头和笔记不是两种数据，是同一份数据的两种视图。**
- 笔记 = 念头的"文字详情页"（展开态）
- 念头 = 笔记的"空间坐标+精简摘要"（折叠态）
- 切换只是视图变化，不是数据迁移，这样才能保证"记录"和"整理"不割裂——用户在整理结构时，点开一个念头写两句，本质就是在记笔记；反过来写笔记时随手把一句话拖出去，就自动成了一个念头节点。

### 1.2 空间不分层，而是分"缩放层级"（Zoom-based，而非 Tab 切换）

这是解决你说的"高度成型结构可视化的同时又不与整个操作空间分裂"的关键手法：**用无限画布 + 缩放语义，而不是用多个独立页面**。

| 缩放层级 | 呈现内容 | 类比 |
|---|---|---|
| 最远景 | 所有"成型结构"作为星系/大陆漂浮，彼此有引力线 | 星图/世界地图 |
| 中景 | 单个结构内部的念头节点与连线，可拖拽重组 | 星系内的星系统/关系网 |
| 近景 | 单个念头展开为笔记详情，可编辑正文 | 单个星球的地表 |
| 最近景（冥想态） | 单个念头以外去除一切UI，只剩呼吸感的一个光点 | 静观 |

用户从任何一层滚轮/捏合缩放，都是**同一个连续坐标系**，没有"跳转页面"的割裂感。这是游戏化和"不分裂"这两个诉求能同时满足的技术核心。

### 1.3 念头的"物理属性"（这是让它像游戏而不是像流程图的关键）

念头不只是一个卡片，要给它类似游戏实体的属性，才会有"操作快感"和"生命感"：

- **质量/引力**：被引用、被连接越多，节点越"重"，视觉上更大、吸附力更强
- **温度/新鲜度**：新念头发光/跳动，久置未碰的念头逐渐"冷却"（颜色变灰、体积收缩，但不删除——对应"觉察到自己哪些念头在自然消退"）
- **张力**：两个念头被强行拖到一起但语义不通，连线会"绷紧/震动"，提示用户重新考虑关系（轻度游戏反馈，不做强制阻拦）
- **聚合体积**：多个念头聚合成结构后，整体有一个"结构强度"分数（连接密度、内部一致性），高分结构才会"结晶"为可视化的固定形态

## 二、功能模块清单

### 模块 A：笔记记录（输入层）

1. **快速捕获**：全局快捷键/小组件/语音输入，随时甩一条念头进空间，不打断心流
2. **自由笔记编辑**：支持长文本、markdown、图片、链接、待办语法（如 `[]` 自动转 todo）
3. **笔记 → 念头自动拆解**：
   - 手动模式：用户自己框选一段文字"甩出"成一个新念头节点
   - 辅助模式（可选AI）：对长笔记做语义分段建议，提示"这几句话可能是独立的念头，是否拆分？"——**建议永远是提示，不自动执行**，避免用户失去掌控感
4. **来源保留**：每个念头都保留指向原始笔记的"锚点"，点开念头可以看到它是从哪条笔记的哪一句话生成的（避免碎片化后失去上下文）

### 模块 B：念头空间（核心画布 / 操作层）

5. **无限画布**：支持平移、缩放、多点触控/鼠标滚轮
6. **拖拽与吸附**：
   - 念头靠近念头，出现"引力提示线"，松手可选择"连接/合并/仅靠近不连接"
   - 支持多选拖拽（框选一片念头一起移动）
7. **连接类型（关系语义）**：不是所有连线都一样，至少要有几种基础关系，否则结构没有意义：
   - 因果（A导致B）
   - 并列/归类（A和B属于同一类）
   - 对立/矛盾（A与B冲突，用于呈现内心矛盾时特别有用）
   - 时间顺序
   - 从属（A是B的子项/细化）
8. **自动布局算法**：力导向图（force-directed layout）作为默认排布，用户手动拖拽后可以"锁定位置"，也可以随时"一键复位"回自动布局（避免手动摆放的疲劳）
9. **结构容器（Cluster/Constellation）**：框选一组念头，可"打包"成一个结构容器，容器可折叠为一个大节点（类似文件夹，但是空间化的、可视化的）
10. **多种呈现模板**（同一份数据，不同布局算法）：
    - 星团图（默认，自由联想）
    - 树状图（层级归纳，适合做大纲）
    - 时间线（适合日记/项目复盘）
    - 看板列（适合转成Todo时）

### 模块 C：观察与内观（觉察层）

11. **观察模式（只读凝视）**：进入后画布不可编辑，只能缓慢平移/缩放，配合呼吸引导的视觉节奏（比如背景光晕随呼吸频率明暗），用于"看着自己的念头结构而不去改动它"——这是内观的核心：观察而不干预
12. **念头计时静观**：点选一个念头，进入"聚焦态"，全屏只剩这个念头和一个呼吸圆环，可设定1-10分钟静观计时，结束后可选"记录此刻的感受"（生成新念头，形成闭环）
13. **念头冷却可视化**：如1.3所述，被搁置的念头会自然"降温"，用户在观察模式下能直观看到"哪些念头执念很重（一直很热/很大）"、"哪些已经自然放下（变冷变小）"——这本身就是一种觉察训练
14. **情绪标注（可选，不强制）**：给念头打情绪色温（红/蓝/灰等），长期积累后可在"最远景视图"看到整体情绪分布的星图，类似情绪日记的空间化版本
15. **每日/每周回顾仪式**：系统在固定时间提示"进入回顾模式"——自动带用户飞到本周新增念头最密集的区域，做一次引导式的"漫游观察"（半自动镜头运动，类似游戏过场，但可随时接管）

### 模块 D：结构成型与可视化奖励（游戏化层）

16. **结晶机制**：当一个结构容器内部的连接密度、一致性达到阈值（或用户手动确认"这个想清楚了"），触发"结晶动画"——念头团从松散星团收拢为一个规则几何体/图腾/建筑形态，插入到"最远景"的星图中作为一个永久地标
17. **成长档案（不是等级分数，是"念头生态"的可视化历史）**：
    - 星图会随时间自然生长，用户可以缩放时间轴回看"半年前我的念头空间长什么样"
    - 不建议做传统游戏的等级/积分/连续打卡数字，容易异化为焦虑源；用"生态是否丰茂、结构是否清晰"这种质性反馈代替数值攀比
18. **仪式感反馈**：完成一次整理、一次静观、一次结晶，都给予克制而不打扰的动效+音效反馈（类似《纪念碑谷》的质感，而非手游的浮夸特效）
19. **"念头园丁"隐喻贯穿全局**：所有UI文案/图标围绕"照料一片念头田野"的隐喻，而不是"管理任务列表"的隐喻——这决定了整体调性不会滑向效率工具的冰冷感

### 模块 E：从结构提炼行动（Todo层）

20. **行动萃取**：在任意结构上，用户可以框选几个念头，点"提炼为行动"，系统生成一条Todo，并保留一条"行动锚点连线"指回它所诞生的结构（避免Todo脱离思考语境，变成孤立任务）
21. **Todo呈现方式**：
    - 空间内：Todo念头有独特的视觉标记（比如带小旗帜），且可以有"完成态"动画（比如从星团中"发射出去"消失，象征想法落地）
    - 空间外：也提供一个传统列表视图/看板视图，方便日常执行时不需要在画布里找
22. **任务与念头双向同步**：完成Todo后，对应念头自动进入"冷却/归档"状态（不是删除，是沉淀），呼应"事情做完了，这个念头可以放下了"的心理闭环
23. **提醒与日程**：可选绑定截止时间，接入系统通知，但克制使用，避免和"觉察空间"的松弛感冲突（建议默认弱提醒，用户主动开启强提醒）

### 模块 F：检索与记忆

24. **全局搜索**：文本、标签、关系类型、时间范围
25. **念头间的"意外重逢"**：系统偶尔（不打扰地、低频地）在空间边缘提示"这个念头和三个月前的一个念头很相似，要不要看看？"——制造"温故而知新"的联想惊喜，这是很游戏化也很有内观价值的机制
26. **标签与自由分类**：允许多标签，不强制单一归类（念头允许属于多个结构）

### 模块 G：数据与隐私

27. **本地优先 + 可选云同步**：笔记内容私密性强，建议本地存储为主，云同步作为可选项，端到端加密
28. **导出**：支持导出为 Markdown/图片(结构快照)/JSON(结构化数据备份)，避免数据锁死

## 三、交互与视觉设计原则（保证"不分裂"的关键约束）

1. **一个坐标系，一个连续缩放，杜绝跳转式页面切换**——所有模式（记录/整理/观察/Todo）都是同一空间不同缩放/滤镜状态
2. **克制的游戏化**：用有机的、自然生长的隐喻（星团、生态、结晶）而不是竞技类隐喻（等级、积分、排行榜），因为这是一个面向内观的工具，过度游戏化会喧宾夺主
3. **编辑与观察用同一套视觉语言，只是交互权限不同**：观察模式下画布长得和整理模式几乎一样，只是不能拖动/编辑，靠"呼吸感"的动效区分状态，而不是靠切换完全不同的UI皮肤
4. **一切自动化建议都是"提示"而非"执行"**：AI辅助拆分笔记、AI推荐关系、AI提醒相似念头，全部以"半透明幽灵预览"形式出现，用户确认才生效——这对"觉察"类工具尤其重要，用户要始终感觉这是"我自己的念头"，不是被系统重构的

## 四、技术架构建议

### 4.1 数据模型（核心Schema）

```
Note {
  id, content(rich text), createdAt, updatedAt, sourceType
}

Thought {
  id, noteAnchorId(指向Note中的片段), 
  position{x,y,zoomLayer}, 
  mass(引力值), temperature(新鲜度,随时间衰减函数),
  colorTag(情绪标注), 
  status: active | cooling | archived | crystallized
}

Edge {
  id, fromThoughtId, toThoughtId, 
  relationType: cause | parallel | conflict | sequence | subordinate,
  strength
}

Structure(Cluster) {
  id, thoughtIds[], edgeIds[],
  cohesionScore(结构强度,决定是否可结晶),
  crystallizedForm(结晶后的几何形态ID), 
  layoutType: force | tree | timeline | kanban
}

Action(Todo) {
  id, sourceThoughtIds[], title, dueDate?, status, 
}
```

### 4.2 前端技术选型

- **渲染层**：推荐 Canvas/WebGL 方案（如 `PixiJS` 或 `Konva.js`）承载"无限画布+大量节点+物理效果"，而不是纯DOM（DOM在几百个节点+物理动画时性能会明显下降）
- **力导向布局**：`d3-force` 非常成熟，可直接用于念头的自动排布、引力吸附效果
- **动效**：`Framer Motion`（如果部分UI仍是DOM/React）+ 自定义 WebGL shader 做"结晶""呼吸光晕"等有机质感效果
- **状态管理**：图结构数据建议用类似 `Yjs`（如果要做多端同步/未来协作）或简单的本地图数据库（`IndexedDB` + 内存图结构，比如用 `graphology` 库管理图运算）

### 4.3 后端/存储

- **本地优先架构**：`SQLite`（桌面/移动）或 `IndexedDB`（Web）存储 Note/Thought/Edge/Structure/Action 五张表
- **同步**：如果要多端同步，推荐 CRDT 方案（Yjs 或 Automerge），天然适合"图结构+离线优先+最终一致"的场景
- **AI辅助功能**（笔记拆分建议、相似念头提醒、关系推荐）：
  - 可用 Embedding（如本地小模型或调用API做文本向量化）计算念头相似度，支撑"意外重逢"机制
  - 笔记拆分建议可用LLM做语义分段，输出建议而非直接执行

### 4.4 性能关键点

- 念头数量达到大规模（几千个）后，需要做**视口裁剪（viewport culling）**：只渲染当前可见区域内的节点，远处的结构用"低模/图标化"聚合展示（类似地图的LOD分层加载）
- 物理模拟（引力、张力）应做节流/异步计算，避免拖拽卡顿

## 五、分阶段实施路线图

### Phase 0：概念验证（2-4周）
目标：验证"缩放式连续空间"这个核心交互是否成立，不做任何AI/游戏化。
- 无限画布 + 基础节点拖拽
- 笔记转念头（纯手动）
- 简单连线（先不分关系类型）
- 力导向自动布局

**里程碑判断**：用户拖10-20个念头形成一个结构，体验是否"顺手""有掌控感"。这是后续一切的地基，做不好后面全白搭。

### Phase 1：MVP（1-2个月）
在Phase 0基础上加：
- 关系类型（至少因果/并列/对立三种）
- 结构容器（打包/折叠）
- 观察模式（只读+呼吸光晕）
- 基础Todo萃取（结构→行动，简单列表视图）
- 本地存储持久化

**目标**：做到"记录-整理-觉察-行动"四个环节能闭环跑通一次，哪怕视觉很糙。

### Phase 2：游戏化与觉察深化（2-3个月）
- 念头温度/冷却机制
- 结晶动画与最远景星图
- 念头计时静观模式
- 情绪色温标注
- 每周回顾仪式（引导式漫游）

### Phase 3：智能辅助（2-3个月，可与Phase2并行）
- AI笔记拆分建议（幽灵预览式）
- 相似念头"意外重逢"提醒
- 关系类型智能推荐

### Phase 4：多端与协作（视需求，3个月+）
- 云同步（CRDT）
- 移动端适配（触控手势重新设计缩放/拖拽）
- （可选）结构分享/多人协作念头空间

## 六、几个需要你提前决策的关键分歧点

这几个问题会显著影响架构，建议现在就想清楚方向，而不是开发中途改：

1. **AI辅助的介入程度**：完全手动（用户自己拆笔记、自己判断结构）vs 半自动建议 vs 高度自动化？—— 建议做成"默认关闭AI，用户可选开启"，保护"这是我自己的觉察"这个核心体验。
2. **游戏化的力度**：要不要有明确的"数值成长系统"（比如念头总数、结晶数量做成可炫耀的成就）？—— 个人建议**不要做排行榜/社交攀比类机制**，这和内观冥想的调性冲突，容易异化用户动机。
3. **单人工具 or 未来协作**：如果完全不考虑协作，数据模型和同步架构可以大幅简化（不需要CRDT，本地SQLite即可）。建议先按纯单人工具设计，协作作为远期可选项，不要一开始就为分布式一致性增加复杂度。
4. **平台优先级**：桌面（更适合复杂空间操作，鼠标精度高）还是移动（更适合随手记录）？—— 建议**记录端优先移动/轻量**，**整理和观察端优先桌面/大屏**，两端通过同一份数据同步，但交互设计要分别优化，不要用同一套UI硬套两端。

## 七、一句话总结这个产品的灵魂

**用户不是在"使用"一个工具管理任务，而是在"照料"一片属于自己的念头田野**——记录是播种，整理是耕作，观察是静观其生长与凋零，行动是收获。所有功能设计都应该回到这句话上做取舍：任何让用户感觉"在被系统管理/评分"的功能，都应该被砍掉或弱化。
