# 念头空间——知识库架构与采集配置规划（Part 5）

> 承接 Part 4 的自主开发闭环。本篇解决："Planner/Coder/Verifier 等子agent检索时，知识库里到底该有什么、怎么组织、怎么采集、怎么保鲜。"没有这一层，Part4的RAG检索只是一句空话。

---

## 一、知识库总体架构：两大类、四个子库

```
┌─────────────────────────────────────────────────────────────┐
│  内部知识库（Internal KB）—— 项目自己产生的、独一无二的知识        │
│  ├─ KB-1 产品与架构规划库（Part1-4文档 + 后续所有设计决策）        │
│  ├─ KB-2 代码库本身（源码 + 接口签名 + 类型定义）                 │
│  └─ KB-3 项目历史与决策记录（ADR、commit历史、失败尝试记录）        │
├─────────────────────────────────────────────────────────────┤
│  外部知识库（External KB）—— 技术生态里已经存在的公共知识          │
│  └─ KB-4 依赖库/框架/算法/合规类文档                             │
└─────────────────────────────────────────────────────────────┘
```

**核心原则**：内部知识库的优先级永远高于外部知识库。当两者冲突（比如d3-force官方文档的默认用法，和本项目架构规划里对物理模拟的特殊约束不一致），agent必须以内部知识库（尤其KB-1的架构约束）为准——这条规则要写进检索排序逻辑，而不是靠agent自己判断。

## 二、内部知识库详细清单（KB-1、KB-2、KB-3）

### KB-1：产品与架构规划库

| 内容 | 来源 | 采集方式 |
|---|---|---|
| Part1 产品功能规划 | 本次对话产出的4份文档 | 直接导入，按标题层级（一、二、三…）切分chunk |
| Part2 几何结构系统规划 | 同上 | 每种几何结构（G1-G12）单独成chunk，保留"数据需求/布局算法/交互细节/视觉规格/触发时机/转出路径"六个字段结构化标注，方便Coder子agent按需精确检索某一种结构的规格 |
| Part3 技术架构规划 | 同上 | SQL schema、TypeScript接口定义单独切chunk（代码块不能和说明文字混在一起切，否则检索时容易只召回一半） |
| Part4 自主开发模式规划 | 同上 | 主要供"元层"使用——即agent循环自己在决定"这个task该不该转人工"时参考 |
| **架构决策记录（ADR）** | 项目推进过程中新产生的决策，需要新建 | 每次做出一个不在原规划中的架构决策（比如"决定用Qdrant而非Chroma"），必须补写一条ADR：背景/选项/决策/后果，格式统一模板，持续追加进KB-1 |
| **接口契约文档** | 从Part3的 `IGeometryEngine` 等接口定义自动生成 | 用 TypeDoc 之类工具从源码注释自动抽取生成文档并写回KB-1，保证"文档"和"代码"不会长期不同步 |

### KB-2：代码库本身

这是最容易被忽视但最重要的知识源——**代码即知识库**，而不是只有Markdown文档才算知识。

| 内容 | 采集方式 |
|---|---|
| 全部源码文件 | 按AST（抽象语法树）做代码感知的切分（code-aware chunking），而不是按固定字符数切——一个函数/一个class应该完整地在一个chunk里，不能从中间切断 |
| 函数/接口签名索引 | 单独抽取所有 `export function` / `export interface` / `export class` 的签名+JSDoc注释，做成一个高优先级的"速查表"子集合，供Coder子agent做"我要调用这个函数，参数是什么"这类高频查询 |
| 已有测试用例 | 单独索引，Coder子agent写新代码前应先检索"类似功能有没有已存在的测试模式可以参考"，保持测试风格一致 |
| 依赖关系图谱 | 用 `dependency-cruiser` 的输出结果生成模块依赖关系的结构化描述，写入KB-2，供Planner子agent判断"改这个文件会不会波及不该碰的模块" |

**更新机制**：KB-2必须做到**代码一变更就重新索引**（而非定期批量刷新），否则agent检索到的是过时代码，会产生"修复了一个已经不存在的bug"这类荒谬结果。用 git pre-commit/post-commit hook 触发增量重新索引。

### KB-3：项目历史与决策记录

| 内容 | 价值 |
|---|---|
| Git commit历史（含commit message） | 让agent能查到"这个模块上次是谁、为什么这么改的" |
| **失败尝试记录（这条最容易被忽略，但价值很高）** | Part4提到验证子agent会有重试，重试失败的尝试不应该丢弃，而应该记录"方案X为什么失败"写入KB-3，避免同一个坑被反复踩——本质是给agent一份"错题本" |
| Phase checkpoint的人工反馈记录 | 人工在Phase边界审阅时提出的修改意见，结构化记录后写回KB-3，作为后续同类task的经验参考 |
| Issue/Bug追踪记录 | 已知问题、已修复问题的对照，避免agent"重新发现"并"重新修复"同一个bug导致代码抖动 |

## 三、外部知识库详细清单（KB-4）

这一层的关键是**不要什么都抓，只抓项目实际依赖到的、且容易和agent的训练知识过期的部分**。开源代码模型的训练知识对新版本库的API变化经常滞后，这是外部知识库存在的核心意义——补足"模型不知道的最新用法"，而不是重复模型已经知道的通识。

### 3.1 直接依赖库/框架文档（对应Part3的技术选型）

| 类别 | 具体来源 | 采集范围建议 |
|---|---|---|
| 数据层 | Yjs官方文档、y-indexeddb、y-websocket文档 | 全量抓取API文档，重点是CRDT数据类型（Y.Map/Y.Array）的边界行为说明 |
| 渲染层 | PixiJS官方文档与API参考、pixi-viewport文档、react-native-skia文档 | 全量API文档 + 官方示例代码仓库（示例代码对agent"照猫画虎"的参考价值往往比纯文字文档更高） |
| 布局算法 | d3-force、d3-hierarchy 官方文档与源码注释 | 全量 |
| 桌面壳 | Tauri官方文档（尤其自动更新、权限模型、打包签名部分） | 全量，且需重点标注版本号（Tauri大版本间API差异较大，必须明确项目锁定的版本） |
| 移动端 | React Native官方文档、react-native-gesture-handler文档 | 全量 |
| 本地存储 | SQLite官方文档、better-sqlite3（或所选Node绑定库）文档 | 全量 |
| AI/Embedding | transformers.js文档、所选sentence-embedding模型的模型卡（model card） | 全量 |
| 工程化工具 | pnpm workspace文档、Turborepo文档、dependency-cruiser规则配置文档、Vitest/Playwright文档 | 全量 |
| **Agent框架自身文档（Part4新增的技术栈）** | OpenHands文档、Aider文档、MetaGPT/CrewAI文档、vLLM部署文档、Qwen2.5-Coder使用指南 | 全量，这部分尤其要保鲜，因为agent工具链本身迭代速度很快 |

### 3.2 算法与设计模式参考（非某个库专属，但项目强依赖）

- 力导向图布局的经典论文/技术博客（Fruchterman-Reingold算法说明），帮助agent理解d3-force参数调优的原理而非只会抄示例代码
- CRDT原理性资料（Yjs背后的理论基础），帮助agent在遇到边界情况时能推理而非死记硬背API
- Canvas/WebGL性能优化最佳实践（视口裁剪、LOD分级的通用实现模式，不限于PixiJS）

### 3.3 合规与安全类参考

- GDPR/CCPA关于数据可携带权、被遗忘权的具体条款摘要（供涉及数据导出/删除功能的task参考）
- 常见前端/桌面应用安全清单（OWASP相关条目），供涉及文件加密、剪贴板权限的task参考

## 四、知识库的组织与索引配置

### 4.1 向量数据库选型确认

沿用Part4建议的 **Qdrant**（相较Chroma，Qdrant在过滤检索/元数据管理上更成熟，适合本项目"多子库+按角色路由"的复杂检索需求）。

### 4.2 Collection划分方案

```
qdrant collections:
├── kb1_product_architecture     # Part1-4文档 + ADR + 接口契约
├── kb2_codebase                 # 源码AST chunk + 函数签名速查表
├── kb2_tests                    # 已有测试用例
├── kb3_project_history          # commit历史 + 失败尝试记录 + checkpoint反馈
├── kb4_dependency_docs           # 外部库官方文档
├── kb4_algorithms                # 算法与设计模式参考
└── kb4_compliance                 # 合规安全类参考
```

分开建collection而不是全塞进一个大库的原因：不同子库的**检索权重和更新频率完全不同**（KB-2要求实时更新，KB-4大部分只需周更），分开管理便于分别设置索引刷新策略，也便于第五节的"按agent角色路由检索范围"。

### 4.3 元数据（Metadata）标注规范

每个chunk写入时必须携带的元数据字段：

```json
{
  "source_collection": "kb2_codebase",
  "source_path": "packages/geometry-engines/tree/index.ts",
  "content_type": "code | doc | adr | test | commit_log | failed_attempt",
  "module": "geometry-engines.tree",
  "version_hash": "对应的git commit hash或库版本号",
  "last_updated": "ISO时间戳",
  "authority_level": "internal_binding | internal_reference | external_reference",
  "language": "zh | en"
}
```

`authority_level` 是关键字段：内部规划文档中明确写死的架构约束（如"core-domain不得依赖渲染库"）标记为 `internal_binding`（检索排序最高，且在冲突时优先采信），一般性说明标记为 `internal_reference`，外部文档一律 `external_reference`。

### 4.4 Chunk切分策略（分内容类型）

| 内容类型 | 切分策略 |
|---|---|
| Markdown规划文档 | 按标题层级语义切分，保留标题路径作为chunk的上下文前缀（如"Part2 > 三、G5矩阵 > ①数据需求"），避免脱离上下文后语义丢失 |
| 源代码 | AST感知切分，一个函数/interface/class为最小单位，附带其所在文件的import语句作为上下文 |
| 外部官方文档 | 优先使用文档站点自带的章节结构切分；API参考类文档按"一个函数/一个API"为最小单位 |
| Git commit历史 | 一个commit为一个chunk，包含message+变更文件列表摘要，不含完整diff（diff过长时另存为可选的"展开引用"） |

## 五、检索策略与agent角色绑定

对应Part4的循环角色，明确规定每个角色在每个阶段应该查哪些collection，避免"眉毛胡子一把抓"式的全库检索（既浪费token又降低相关性）：

| Agent角色 | 检索阶段 | 优先查询的collection | 检索方式 |
|---|---|---|---|
| Planner（规划子agent） | 拆解task为文件级计划前 | `kb1_product_architecture`（架构约束）+ `kb2_codebase`（现有代码结构） | 先查KB-1确认该task涉及的模块有没有明确的架构约束，再查KB-2看现有代码怎么组织的 |
| Coder（编码子agent） | 生成代码前 | `kb2_codebase`（函数签名速查表）+ `kb4_dependency_docs`（涉及的外部库用法）+ `kb2_tests`（参考测试风格） | 优先精确检索"我要调用的这个接口的签名和用例"，而非泛泛检索整个模块 |
| Coder（遇到不确定的库用法时） | 编码中 | `kb4_dependency_docs` + `kb4_algorithms` | Hybrid search（向量检索+关键词BM25结合），因为库API名称这类精确术语用关键词检索往往比纯向量检索更准 |
| Verifier（验证子agent） | 测试失败分析时 | `kb3_project_history`（尤其"失败尝试记录"子集） | 检索"这个报错之前有没有出现过类似的失败记录"，避免同一个坑反复踩 |
| Committer（提交子agent） | 生成commit message时 | `kb1_product_architecture`（对应哪个Phase哪个task） | 保证commit message能追溯到规划文档里的具体任务编号 |
| 人工Phase Checkpoint审阅辅助 | 生成阶段性总结报告时 | 全部collection做综合检索 | 用于自动生成"本阶段完成了什么、遇到了什么问题"的checkpoint简报，减少人工阅读原始日志的负担 |

## 六、知识库更新与维护机制

| 知识库 | 更新触发方式 | 更新频率 |
|---|---|---|
| KB-1 产品架构库 | 人工新增/修改规划文档时手动触发重新索引；ADR每次新增立即索引 | 事件触发 |
| KB-2 代码库 | git commit hook 自动触发增量索引（只重新索引变更的文件） | 实时（每次commit） |
| KB-2 测试用例 | 同上 | 实时 |
| KB-3 项目历史 | commit历史自动追加；失败尝试记录由Verifier子agent在判定失败时自动写入 | 实时 |
| KB-4 依赖库文档 | 定时任务检查所依赖库的版本号，若发现锁定版本更新，触发重新抓取对应文档 | 周期性检查（建议每周），版本锁定文件（如package.json的精确版本）变更时立即触发 |
| KB-4 算法/合规类 | 基本静态，人工判断是否需要更新 | 低频（按需） |

**过期知识清理**：KB-4的外部文档在库版本升级后，旧版本文档不应直接删除，而是标记 `deprecated: true` 并降低检索权重，保留供排查"历史代码为何这样写"时参考，但不作为新代码生成的首选依据。

## 八、风险与质量控制

| 风险 | 应对 |
|---|---|
| 外部文档抓取到过时/错误的第三方博客内容，污染知识库 | 外部抓取源优先限定为**官方文档站点和官方GitHub仓库**，非官方博客/教程类内容不纳入自动抓取流程，如需引用需人工审核后手动录入并标注来源可信度 |
| 内部KB-1文档和实际代码逐渐脱节（文档没跟上代码变化） | 第二节提到的"接口契约文档自动从源码生成"机制是关键防线；此外建议每个Phase checkpoint时人工顺带检查一次KB-1是否需要补充ADR |
| 知识库过大导致检索延迟、上下文塞入过多噪音 | 严格执行第五节"按角色路由检索范围"，禁止任何子agent做无限定范围的全库检索；单次检索返回结果数控制在合理范围（如top 5-8个chunk），而非返回大量低相关度结果 |
| 失败尝试记录（KB-3）本身质量参差不齐，可能误导后续判断 | Verifier子agent写入失败记录时，必须包含结构化字段（报错信息、尝试的方案、失败原因推测），而非随意的自由文本，便于后续检索匹配的精确度 |

## 九、与Part4自主循环的最终对接图

```
Part4的自主循环                          本篇的知识库系统
─────────────────────                   ─────────────────────
任务队列 ──────────────────────────────► 检索 KB-1（该task涉及的架构约束）
   │
   ▼
Planner子agent ─────────────────────────► 检索 KB-1 + KB-2（架构约束+现有代码结构）
   │
   ▼
Coder子agent ───────────────────────────► 检索 KB-2 + KB-4（接口签名+外部库用法）
   │
   ▼
Verifier子agent（测试/lint/依赖规则检查）
   │  失败 ──────────────────────────────► 检索 KB-3（是否有类似失败记录）
   │                                        │
   │  ◄─────────────────────────────────── 写入新的失败记录到 KB-3
   │
   ▼ 通过
Committer子agent ───────────────────────► 检索 KB-1（关联具体task编号生成commit message）
   │
   ▼
[下一个task]，同时触发 KB-2 增量重新索引（因为代码刚刚变更）
```

## 十、这一层规划要回答的问题

知识库不是"喂给agent一堆文档"这么简单，真正的设计难点是：**在正确的时机，把正确颗粒度的知识，路由给正确的agent角色**。如果四份内部/外部知识库不分collection、不分角色地一股脑塞进单一向量库做检索，Part4设计的整个自主闭环会退化成"每次都检索到一堆不相关内容，模型自己再去噪"——这既浪费token，也是开源模型（能力本就弱于顶尖闭源模型）最容易出错的场景。**知识库的组织质量，直接决定了Part4自主开发模式的成功率上限。**
