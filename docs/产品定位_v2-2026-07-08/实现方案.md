## §0 实现策略

### 0.1 核心判断

**这不是在旧代码上的修补，是基于空间交互设计.md的全新实现。**

| 维度 | 旧代码（TAS路径） | 新实现（v2路径） |
|---|---|---|
| 空间模型 | 凸包多面体+6面cube-camera | 有限3D念头晶体+6层+2外置+三维度 |
| 念头模型 | 半透明光球 | 可塑多面体（锐利/圆润）+相变 |
| 操作模型 | 固定6面切换 | 圆周轨道视角+1/4外层滑动 |
| 内容模型 | 纯文字念头 | 行制+块制+行内元素+连线+维度 |
| 关系模型 | edge不可见 | 五种基础关系+规则算法 |
| 定位模型 | 温度→Y轴 | 标签数值+位置算法 |

**结论**: 旧代码的render/topology/sim层全部废弃，core/persistence层部分保留，工程基础设施（check脚本/GEB/Yjs）保留。

### 0.2 三条红线

1. **不破坏旧代码的可运行性** — 新实现与旧代码并存，通过feature flag切换，旧代码保持可运行直到新实现验收通过
2. **不一次性实现所有功能** — 分阶段交付，每个阶段有可验收的闭环
3. **不跳过验收** — 每个阶段必须通过对应验收标准才能进入下一阶段

### 0.3 与现有代码的关系

| 现有代码 | 处理方式 | 理由 |
|---|---|---|
| `src/v2/core/` | **v2新建** | v2新目录，v1的`src/core/`不动 |
| `src/persistence/`（Yjs+IndexedDB） | **保留扩展** | 持久化层不变，需扩展支持新数据结构 |
| `src/render/`（Three.js渲染） | **废弃重写** | 渲染模型完全不同（凸包→晶体+层+相变） |
| `src/topology/`（凸包/cube-camera） | **废弃** | 不再使用凸包和6面切换 |
| `src/sim/`（力导向） | **重构保留** | 3D力导向仍需要，但需适配层约束 |
| `src/ui/`（toolbar） | **重构保留** | 工具栏模式保留，内容重写 |
| `src/runtime/flags/` | **保留扩展** | feature flag系统用于新旧切换 |
| `scripts/`（check脚本） | **保留扩展** | GEB工作流不变，新增v2相关检查 |
| `tests/` | **重构** | 旧测试废弃，新测试随实现编写 |

**check脚本扩展计划（W-11）**: S0需更新以下check脚本以覆盖v2新目录:
- check-arch.mjs: 新增v2目录的arch检查

**瞬态层说明（T-F3）**：动画状态不进 CRDT。Thought 的 Yjs 只存 `type: 'thought'|'memory'`（相变目标态）；
动画状态（t 值、animating 标志）存于内存瞬态层（`Map<thoughtId, AnimationState>`），不进 Yjs，动画插值由渲染层驱动。

**UndoManager 追踪范围（T-F5）**：UndoManager 追踪所有用户可操作 Y.Map：
`thoughts, relations, rules, positionAlgorithms, markAlgorithms, marks, dampers, todos`，保留 `captureTimeout=100ms`。

**观察者策略（W-1）**:
- 顶层 `thoughts.observe`: 监听念头增删，分发到 render 层
- 每个Thought的 `Y.Map.observe`: 监听字段变更（type/layer/position等）
- `tags/attributes` 用 `observeDeep`: 监听嵌套变更
- `content` (Y.XmlFragment) 用 `observe`: 监听文档变更
- bridge层负责分发到render层，禁止render层直接observeDeep整个doc

**content字段说明（W-3）**: content使用Y.XmlFragment支持结构化文档协作编辑。对应§5.11的行制+块制+行内元素+连线+维度结构，Y.XmlFragment的DOM结构映射nx-clip的JSON格式。

**历史快照存储策略（W-2）**: 历史快照不存入Yjs（避免文档无限膨胀），存入独立IndexedDB object store。Yjs仅存当前态。快照通过thought-bridge在创建时序列化到IndexedDB。

**字段补全说明（WARN-10）**:
- `todos: Y.Map<Y.Map>`（Todo实体，对应C-F1）— 已在上方schema中
- `subspaces`: 已在Thought中通过`parentSpaceId`/`subspaces`字段覆盖（对应C-F2）
- 时间视角配置: `userConfig.customTimeViews: Y.Array<Y.Map>`（对应WARN-8）— 已在userConfig中补充
- 空间外部信息边界配置: `spaceConfig.boundary: Y.Map`（对应C-F3）— 已在spaceConfig中补充

## §4 风险与缓解

| # | 风险 | 严重度 | 缓解 |
|---|---|---|---|
| 1 | Three.js相变动画性能 | 🟡 P1 | 分两阶段（几何先，内容后），用插值着色器 |
| 2 | 空间重组算法复杂度 | 🟡 P1 | 力导向+防抖，限制重组频率 |
| 3 | 内容格式编辑器复杂度 | 🔴 P0 | S6分两步：先基础块制，后连线维度 |
| 4 | 规则引擎表达能力 | 🟡 P1 | 先图形化，脚本和AI辅助后续补 |
| 5 | 移动端3D性能 | 🟡 P1 | InstancedMesh+LOD+上限管理 |
| 6 | 新旧并存的技术债 | 🟡 P1 | 旧代码标记deprecated，v2验收通过后删除 |
| 7 | 服务端开发工作量 | 🟡 P1 | S8可延后，本地优先保证核心体验 |
| 8 | Yjs数据一致性 | 🟡 P1 | S8多设备同步可能因嵌套Y类型操作复杂导致冲突; S8前完整测试嵌套Y.Map/Y.Array并发编辑场景; bridge层添加数据完整性校验 |
| 9 | 移动端手势冲突 | 🟡 P1 | 三指撤销/双指缩放/长按/拖拽/滑动5种手势在3D canvas上互斥判定困难; S3设计手势识别状态机; 区分单指/双指/三指的触控起点和持续时间; 提供手势冲突时的用户反馈 |
| 10 | Yjs↔Three.js同步阻塞 | 🟡 P1 | Yjs observe回调是同步的，高频CRDT update会阻塞主线程; 引入rAF批量更新（dirty flag + requestAnimationFrame合并）; bridge层收集变更，每帧统一应用到render层 |

**相变性能缓解措施补充（W-4）**: (1) S2起引入LOD（远距离念头降面数）; (2) 力导向用Web Worker（d3-force-3d支持worker）; (3) 相变动画限制同屏并发数（最多3个同时相变）; (4) InstancedMesh在S9引入但需评估个体形态差异的影响
