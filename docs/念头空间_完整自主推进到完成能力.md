# 念头空间——完整自主推进到完成能力规划（Part 11）

> 承接 Part 1-10。本篇解决："如何从Part4的'自主执行既定任务清单'，进一步升级到Fable5式的'自主推进到完成'——即agent能自己判断下一步做什么、自己判断是否做完、遇到规格空白自己起草方案，而不是每一步都依赖人事先规划好。"

---

## 一、先厘清一个本质区别

| | Part4已有方案 | 真正的"自主推进到完成" |
|---|---|---|
| 任务来源 | Part3第十一节人工预先拆好的task清单 | agent自己发现"接下来该做什么"，动态生成task |
| 完成判定 | 清单跑完即算完成该Phase | agent自己评估"当前状态离目标还有多远"，判断是否真的达到可交付标准 |
| 遇到规格空白 | 只能执行已写好的，规格没写清楚的地方无法处理 | agent依据项目宪法自主起草规格提案，先推进，标记待复核 |
| 人工介入方式 | 阻塞式checkpoint（必须人工点头才能继续） | 大部分改为事后审阅（agent先做，人工事后抽查/纠偏），只保留真正高风险的阻塞点 |

**Part4方案本质是"有监工在场的自动化流水线工人"，Fable5式方案更接近"有明确目标、被授权自主决策的项目负责人"。** 后者能力更强，但风险也更高——本篇会把新增能力和新增风险一起讲。

---

## 二、需要补齐的核心能力

### 2.1 目标状态机与多层完成判定（Definition of Done）

没有清晰的"完成"定义，自主系统就无法自己判断该不该停。需要三层DoD：

```
【Task级DoD】（已有，即Part8 1.4的验证清单：编译通过+lint通过+测试通过+架构检查通过+文档同构）

【Phase级DoD】（新增，需要显式定义，而非"清单跑完就算完"）
  示例（Phase 1 MVP）：
  - 功能性：笔记记录、星团/树/管道三种几何结构、Todo萃取、观察模式，均可端到端跑通
  - 质量：核心模块测试覆盖率≥85%，无P0/P1级已知bug
  - 性能：60fps@500节点可见、冷启动<2秒（对应Part8性能预算）
  - 体验：至少完成一轮Part8提到的可用性测试，且核心交互（拖拽/连线/结构切换）主观评分达标
  - 文档：GEB协议L1/L2/L3文档同构检查全部通过

【项目级DoD】（新增，"可发布"的最终判定）
  - 全部Phase级DoD达标
  - 安全/隐私/License检查（Part8 1.7）全部完成
  - 灾难恢复、数据备份机制验证通过
  - 至少一次完整的用户可用性测试反馈闭环
```

**关键设计**：Phase级和项目级DoD必须是**可自动检测的量化指标+必须人工确认的主观指标**的组合，agent可以自主判断"量化指标是否达标"，但"主观指标"（比如可用性测试的用户反馈）永远需要人类参与收集，这是自主程度的天然边界之一。

### 2.2 自主任务生成与待办管理（Autonomous Backlog Grooming）

这是从"执行清单"跃迁到"自主推进"的核心能力。新增一个 **Backlog Groomer 子agent**：

```
Backlog Groomer 工作流：
1. 对照当前Phase的DoD（2.1节），扫描现状（跑测试、查覆盖率、检查功能清单）
2. 找出"现状"和"DoD"之间的差距
3. 把每个差距转化为具体的task，评估优先级，加入任务队列
4. 与Planner子agent（Part3已有）交接，进入常规执行循环
```

这个角色和Part3原有的"人工预先写好的task清单"不冲突，而是**接管人工在Part3第十一节做的那份工作，让系统自己持续生成后续task**，人类不再需要每完成一批task就重新规划下一批。

**关键护栏**：Backlog Groomer生成的新task必须先过一道"目标锚定检查"——检查该task是否服务于项目宪法（Part10 3.2节）里的核心理念和当前Phase目标，防止自主生成的任务逐渐偏离产品方向（比如agent因为发现某个边缘性能问题就自主决定"应该做WebGPU重构"这种超出当前Phase范畴的大动作）。

### 2.3 长程会话的持久化与断点续跑

自主推进到完成意味着系统要能连续运行数天甚至更久，必须支持：

- **状态快照**：每完成若干个task，自动生成"当前进度快照"（哪些task完成、哪些进行中、Backlog Groomer当前对DoD差距的评估），写入项目宪法文档的"当前阶段/当前待解决问题"两节（Part9已设计此更新点，这里明确其对长程运行的必要性）
- **断点续跑**：任何原因中断后（人工暂停、系统重启、成本上限触发的暂停），重新启动时先读取最近一次状态快照，而不是从零开始重新评估整个项目现状——这既是效率问题，也是防止agent"忘了之前的判断，重新做出不一致决策"的一致性保障

### 2.4 自主规格补全（Self-Specification）

Part1-10已经是相当完整的规格文档，但真实开发中一定会遇到规格没覆盖到的细节（比如"拖拽两个念头距离多近才触发吸附提示"这种微观参数）。之前的方案默认"规格不清楚就该问人"，但这会导致自主推进频繁被打断。补充机制：

```
遇到规格空白时：
1. 先检索知识库（Part5），确认是否真的没有相关规格（而非检索遗漏）
2. 确认无规格后，依据项目宪法的核心原则（而非凭空猜测）自主起草一个具体方案
   （例如："吸附触发距离"这类交互细节参数，依据"克制而不过度反馈"的产品原则，
   自主设定一个保守的初始值，并标注为待人工校准）
3. 标记该决定为 `self_specified: true`，写入ADR（Part5 KB-1），而不是静默处理
4. 继续推进，不阻塞
5. 所有 `self_specified` 标记的决定，在下一次Phase checkpoint时集中批量供人工审阅确认或调整
   （而不是逐个打断当下的工作流程）
```

**这是本篇最重要的一条新增能力**：把"遇到不确定就停下问人"改成"依据既有原则自主决定+标记+继续推进+事后批量复核"，是真正让自主循环不被频繁打断的关键。

### 2.5 事后审阅架构（Post-hoc Review）替代部分阻塞式Checkpoint

Part4原设计的5个Phase边界checkpoint都是**阻塞式**（agent必须停下等人工点头）。要接近Fable5式的持续推进，需要区分：

| Checkpoint类型 | 处理方式 |
|---|---|
| 涉及核心架构决策、新增几何结构、数据模型变更 | **保留阻塞式**，必须人工事先批准才能继续（高风险，改起来代价大） |
| Phase内常规task的完成确认、`self_specified`参数的校准 | **改为事后审阅**：agent按自己的判断继续推进，人工在系统运行的同时或稍后抽查，发现问题再回滚/调整，不阻塞主流程 |
| 安全/隐私/数据相关的代码改动 | **保留阻塞式**（Part8已强调这类必须人工review，不可自主） |

事后审阅需要配套：**清晰的变更日志+影响范围标注**，让人工能快速定位"这次自主推进都做了什么、哪些是需要重点看的高风险项"，而不是要人工重新审阅全部代码。

### 2.6 自主质量把关升级：探索性测试与边界用例生成

现有Verifier（Part4/8）只跑"已经写好的测试"，真正接近自主完成能力需要agent自己**生成新的测试场景**，尤其是边界情况：

```
新增 Exploratory Tester 子agent：
- 对每个新完成的模块，基于其接口定义（IGeometryEngine等），自动生成一批边界输入
  （空输入、超大规模输入、并发操作、异常格式数据）
- 跑这些自生成的测试，发现原有测试集未覆盖的问题
- 发现问题后，把这些新用例正式补充进测试套件（而不是只是临时跑一次就丢弃）
```

这个角色让测试覆盖率不再完全依赖人工/Coder子agent当时想到了什么，而是有一个专门角色持续"找茬"，是提升自主完成质量下限的关键。

### 2.7 自主发布与上线闭环

若要真正"推进到完成"而不止是"推进到代码写完"，还需要覆盖：

```
Release Manager 子agent 职责：
1. 项目级DoD（2.1节）全部达标后，自动执行构建打包（对应Part3 CI/CD配置）
2. 生成版本发布说明（基于本次迭代的commit历史与task清单自动汇总，而非人工手写）
3. 触发签名/上传（Tauri自动更新机制，Part8已提及）
4. 发布后接入错误监控（Part8提到的Sentry类工具），持续观察上线后是否有新增异常
5. 若上线后错误率超过预设阈值，自动触发回滚流程并升级人工介入（这是唯一允许"自主执行生产环境回滚"的场景，因为止损的紧迫性高于"等人工确认"的必要性）
```

**发布这一步建议保留人工最终确认**（即便DoD全部达标，"要不要现在发这个版本"仍是商业/时机判断，不适合完全自主决定），但发布前的准备工作（打包、生成说明、跑最终检查）可以完全自主完成，大幅压缩人工需要参与的时间。

### 2.8 自我改进闭环的度量化

Part7设计的"知识蒸馏飞轮"目前是定性描述（"应该反哺知识库"），要真正支撑长期自主运行，需要加上量化反馈：

```
持续追踪的指标（写入一个"系统健康度仪表盘"，而非只存在于日志里）：
- task一次通过率（不需要重试就通过验证的比例）——反映知识库/prompt质量
- self_specified决定被人工推翻的比例——反映agent自主决策的可信度
- 平均每个task的重试次数——反映是否需要针对性优化某类task的prompt
- Insight Ledger中"转正"为可信知识的比例——反映顿悟机制产出的实际价值

这些指标应该按模块/task类型分别统计，用于动态调整：
- 哪些模块可以进一步放宽自主程度（历史通过率高、self_specified决定很少被推翻）
- 哪些模块需要收紧（重试率高、决定经常被推翻），临时改回更保守的阻塞式人工介入
```

**这是让"自主程度"本身变成一个可以基于真实数据动态调整的参数，而不是一次性设定后就不变。**

### 2.9 自主成本/资源调度

长程自主运行必须自己管理资源分配，而不是每次都用同一个模型跑所有task：

```
Resource Scheduler：
- 简单/高确定性task（样板代码、已有相似案例的模块）→ 用更小/更快的模型（如Qwen2.5-Coder-7B级别）
- 复杂/架构性task → 用更大模型（32B+级别）甚至触发Part7的多模型交叉校验
- 设置每日/每周成本预算硬上限，接近上限时自动降级到"只做高优先级task"模式，而非平均分配到所有task直到超支
```

## 三、风险再评估：自主程度提升后的新增风险与护栏

| 新增能力 | 新增风险 | 对应护栏 |
|---|---|---|
| 自主任务生成（2.2） | 逐渐偏离产品方向，做出宪法之外的"自我发挥" | 目标锚定检查（2.2已述）+ 每个Phase结束时人工快速浏览"本阶段agent自主生成了哪些task"，而非逐个审批但至少扫一眼全貌 |
| 自主规格补全（2.4） | 大量`self_specified`决定累积后集中发现方向性错误，返工成本高 | 限制`self_specified`只能用于"局部、可逆、影响范围小"的参数级决定，凡涉及数据结构/接口签名的规格空白，仍必须阻塞等待人工（不属于可自主补全的范畴） |
| 事后审阅替代阻塞checkpoint（2.5） | 错误发现得更晚，影响范围可能已经扩散 | 配套完善的变更日志与影响范围标注（2.5已述）+ 保留"随时可回滚到任意快照"的能力（对应Part9的快照机制），把"发现晚"的代价压低到"回滚成本可控"的范围 |
| 自主发布闭环（2.7） | 生产环境自主操作的风险等级最高 | 发布本身保留人工最终确认；仅"上线后错误率超阈值的自动回滚"这一单一场景允许自主执行，且回滚后必须升级人工介入，不允许自动重新发布 |
| 自主成本调度（2.9） | 为省成本而降级模型可能导致关键task质量下降却未被察觉 | 模型降级仅适用于"历史通过率数据证明该类task用小模型也能稳定通过验证"的场景（依据2.8节的量化指标），不是无差别降级 |

## 四、分阶段实现路径：从Part4现状过渡到完整自主推进

不建议一步到位实现本篇全部能力，建议：

1. **第一步（已在Part4-9设计完成）**：先跑通"人工预先规划task清单+自主执行+阻塞式checkpoint"，让基础闭环稳定运行至少完成Phase 0-1
2. **第二步**：引入Backlog Groomer（2.2）+ 状态快照持久化（2.3），让系统在**单个Phase内部**能自主生成和排布task，但Phase边界仍保留阻塞式人工确认
3. **第三步**：引入自主规格补全机制（2.4）+ 把部分checkpoint改为事后审阅（2.5），扩大自主推进的连续性
4. **第四步**：引入探索性测试（2.6）+ 度量化的自我改进闭环（2.8），提升自主运行下的质量下限和可持续优化能力
5. **第五步**：引入自主发布闭环（2.7）+ 自主成本调度（2.9），此时系统已经接近"从任务生成到代码到测试到发布"的完整自主推进能力

**每一步过渡前，都应该用上一步积累的第2.8节量化指标数据作为"是否可以放心进入下一步"的判断依据**，而不是凭计划表上的时间点机械推进。

## 五、一个诚实的边界说明

即便完整实现本篇全部能力，仍然建议保留几个**不应该自主化**的环节，这不是能力不够，而是原则性判断：

- **是否发布一个新版本给真实用户**——这是商业/时机判断，agent可以把一切准备工作做完，但"现在要不要发"应该始终是人的决定
- **涉及用户隐私/数据安全的代码变更**——即便自主循环的通过率再高，这类改动的潜在后果不对称地严重（一旦出错影响的是用户信任），值得保留人工阻塞式审查
- **架构性决策（新增几何结构、修改核心数据模型）**——这类决策的影响面广、逆转成本高，"自主生成task"和"自主决定要不要做这类task"应该分开，后者永远需要人工拍板

**"自主推进到完成"的目标不是让人完全退出，而是把人的参与从"逐行盯着流程走"压缩到"在真正需要判断力、需要为后果负责的少数关键点上出现"。** 这是本篇设计的自主程度上限，也是它和"完全无人值守、什么都不管"之间的本质区别。

## 六、这一层规划要回答的问题

真正决定这套系统能不能达到"自主推进到完成"的，不是技术能力堆得多全，而是：

> 我愿意把"完成"的判断权，交给系统到什么程度？

第一节的三层DoD体系，就是在把这个抽象问题变成可以逐步验证、逐步放权的具体机制——先让系统证明它能准确判断"task完成了没有"，再逐步验证它能准确判断"Phase完成了没有"，最后才谈得上让它自主判断"整个项目是不是可以推进到发布"。这个信任积累的过程，本身就应该是第四节分阶段路径里最核心的验证逻辑，而不是一次性把权力全部下放。
