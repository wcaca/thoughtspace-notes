## 一、总体技术选型确认

### 1.1 跨端策略的核心矛盾

这个产品有三个诉求会互相拉扯：
- **画布渲染要极致流畅**（大量节点+物理动画+多几何结构），倾向用 WebGL/Native 渲染
- **要多端（桌面/移动/Web）覆盖**，倾向用跨平台框架复用代码
- **业务逻辑复杂**（12种几何算法、结构识别、CRDT同步），倾向把逻辑和UI彻底解耦，逻辑层与平台无关

**结论：采用"逻辑层完全平台无关 + 渲染层按平台适配"的架构**，而不是指望一套UI代码跑所有端。

### 1.2 具体选型

| 层 | 选型 | 理由 |
|---|---|---|
| 核心业务逻辑 | **TypeScript**，纯逻辑无UI依赖 | 可以被 Web / Tauri(桌面) / React Native(移动) 三端同时引用，一份代码 |
| 数据层 | **Yjs (CRDT)** 作为内存中的唯一数据源（single source of truth） | 天然支持离线优先、结构化图数据、后续多端同步；即使暂不做协作，也用它管理本地状态的响应式更新和撤销/重做 |
| 本地持久化 | **SQLite**（桌面/移动）+ **IndexedDB**（Web，via `y-indexeddb` 做 Yjs 持久化） | 两端行为一致：Yjs文档定期snapshot写入对应的本地存储 |
| 桌面壳 | **Tauri**（而非 Electron） | 包体小、内存占用低、Rust后端可以承担性能敏感计算（如力导向布局的物理模拟可选下沉到Rust） |
| Web端 | **React + Vite** | 生态成熟，方便与Tauri共享大部分前端代码（Tauri本质是套壳的Web视图） |
| 移动端 | **React Native** | 与Web共享业务逻辑层，UI层单独适配移动手势（缩放/拖拽在触屏上的交互模型不同，不能照搬桌面） |
| 画布渲染引擎 | **PixiJS**（Web/桌面），**react-native-skia**（移动） | 两套渲染实现，但都基于同一份"场景描述数据"（渲染层只负责把领域模型画出来，不包含业务逻辑） |
| 力导向布局计算 | **d3-force**（前期，纯JS即可） | 后期如果节点规模上千导致卡顿，可评估用 Rust + WASM 重写物理模拟部分 |
| AI/Embedding（螺旋结构、相似念头提醒） | **本地小模型优先**：`transformers.js` 运行轻量 sentence-embedding 模型；云端API作为可选增强 | 保护隐私（笔记内容敏感），且离线可用 |
| 状态管理（UI层） | **Zustand**（订阅 Yjs 文档变化，映射为React可用的响应式状态） | 轻量，避免Redux的模板代码开销 |
| 动效 | **GSAP** 驱动 Pixi 对象的 tween 动画；React端UI动效用 **Framer Motion** | 分工明确：画布内容动效走GSAP+Pixi，画布外UI走Framer Motion |

## 二、总体分层架构

```
┌─────────────────────────────────────────────────────────┐
│  Presentation Layer（按平台分别实现）                       │
│  - Web/Desktop: React + PixiJS Canvas                    │
│  - Mobile: React Native + react-native-skia               │
└───────────────────────┬───────────────────────────────────┘
                         │ 订阅状态 / 派发用户操作指令
┌───────────────────────▼───────────────────────────────────┐
│  Interaction Layer（平台相关，但状态机定义平台无关）           │
│  - 拖拽状态机 / 缩放手势识别 / 长按菜单                        │
│  - 输出统一的 Intent（如 MoveThought, CreateEdge, SwitchGeometry）│
└───────────────────────┬───────────────────────────────────┘
                         │ dispatch(Intent)
┌───────────────────────▼───────────────────────────────────┐
│  Domain / Business Logic Layer（纯TS，平台无关，核心资产）     │
│  ├─ NoteService        （笔记CRUD、笔记→念头拆解）             │
│  ├─ ThoughtService     （念头CRUD、温度衰减计算、质量计算）      │
│  ├─ StructureService   （结构容器、结晶判定）                  │
│  ├─ GeometryEngine     （12种几何算法的统一接口与实现，见第五节）│
│  ├─ RecommendationEngine（结构识别推荐、相似念头检测）          │
│  ├─ ActionService      （Todo萃取、状态流转）                  │
│  └─ ReviewRitualService（每周回顾、静观计时逻辑）               │
└───────────────────────┬───────────────────────────────────┘
                         │ 读写
┌───────────────────────▼───────────────────────────────────┐
│  Data Layer                                               │
│  - Yjs Document（内存中的响应式唯一数据源，含撤销栈）           │
│  - Persistence Adapter（Yjs ⇄ SQLite / IndexedDB）          │
│  - Sync Adapter（可选，Yjs ⇄ 远程同步服务，见第九节）           │
└─────────────────────────────────────────────────────────┘
```

**关键原则**：Domain Layer 不知道自己跑在Web、桌面还是移动端，也不知道渲染用的是Pixi还是Skia——它只操作 Yjs 文档、暴露纯函数式的 Service 接口。这样12种几何算法、结构识别、温度衰减等核心逻辑只写一次，三端复用。

## 四、本地文件体系与数据组织

### 4.1 应用数据目录结构（桌面/移动端落盘）

```
~/.thoughtspace/                       # 或平台标准应用数据目录
├── db/
│   └── thoughtspace.sqlite            # 主数据库（结构化元数据）
├── yjs-snapshots/
│   └── doc-snapshot-{timestamp}.bin   # Yjs文档定期快照（用于崩溃恢复/历史回溯）
├── attachments/
│   ├── images/
│   │   └── {noteId}/{uuid}.jpg
│   └── audio/                         # 语音速记原始文件（若保留原始音频）
├── embeddings-cache/
│   └── {thoughtId}.vec                # 本地embedding向量缓存，避免重复计算
├── exports/
│   └── {date}-export.json / .md / .png
└── logs/
    └── app.log
```

**为什么SQLite + Yjs两套并存而不是只用一个：**
- Yjs 负责"实时协作友好的结构化图数据"（念头、边、坐标），天然支持撤销/重做、增量同步
- SQLite 负责"重查询场景"（全文搜索、按时间范围检索、标签过滤），这类查询用SQL做比在Yjs的内存结构里遍历高效得多
- 两者关系：**Yjs是权威数据源，SQLite是可重建的查询索引**——SQLite表结构可以随时通过重放Yjs文档全量重建，因此SQLite schema变更不需要"迁移旧数据"这么谨慎，只需要"重新建索引"

### 4.2 SQLite 表结构（作为查询索引层）

```sql
CREATE TABLE notes (
  id TEXT PRIMARY KEY,
  content TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  source_type TEXT
);

CREATE TABLE thoughts (
  id TEXT PRIMARY KEY,
  note_anchor_id TEXT,
  mass REAL DEFAULT 1.0,
  temperature REAL DEFAULT 1.0,
  color_tag TEXT,
  status TEXT CHECK(status IN ('active','cooling','archived','crystallized')),
  created_at INTEGER NOT NULL,
  FOREIGN KEY (note_anchor_id) REFERENCES notes(id)
);

CREATE TABLE edges (
  id TEXT PRIMARY KEY,
  from_thought_id TEXT NOT NULL,
  to_thought_id TEXT NOT NULL,
  relation_type TEXT CHECK(relation_type IN ('cause','parallel','conflict','sequence','subordinate')),
  strength REAL DEFAULT 1.0
);

CREATE TABLE structures (
  id TEXT PRIMARY KEY,
  geometry_type TEXT NOT NULL,
  cohesion_score REAL DEFAULT 0,
  crystallized_form TEXT,
  axis_definitions_json TEXT   -- 存储矩阵/光谱的轴定义
);

CREATE TABLE structure_members (
  structure_id TEXT NOT NULL,
  thought_id TEXT NOT NULL,
  PRIMARY KEY (structure_id, thought_id)
);

CREATE TABLE thought_lens_coordinates (
  thought_id TEXT NOT NULL,
  structure_id TEXT NOT NULL,
  geometry_type TEXT NOT NULL,
  coord_json TEXT NOT NULL,     -- {parentId, timelinePosition, axisXValue, ...} 按geometryType使用对应字段

);

CREATE TABLE actions (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  due_date INTEGER,
  status TEXT CHECK(status IN ('todo','doing','done','blocked')),

CREATE TABLE action_sources (
  action_id TEXT NOT NULL,
  thought_id TEXT NOT NULL,

-- 全文检索
CREATE VIRTUAL TABLE notes_fts USING fts5(content, content='notes', content_rowid='rowid');
```

## 五、几何引擎的插件化接口设计

这是整个系统最核心的可扩展点，必须定义清楚统一接口，让12种结构（以及未来可能新增的结构）都能被渲染层、推荐引擎以一致的方式调用。

```typescript
// packages/geometry-engines/types.ts

export interface GeometryCoordinate {
  [key: string]: number | string | undefined;
  // 树: { parentId }
  // 时间线: { timelinePosition, track }
  // 矩阵: { axisXValue, axisYValue }
  // ...按具体geometryType扩展
}

export interface LayoutResult {
  positions: Map<ThoughtId, { x: number; y: number }>;
  edgesToRender: RenderEdge[];
  backgroundHints: BackgroundGridSpec; // 告知渲染层该画什么背景网格
}

export interface IGeometryEngine {
  type: GeometryType;

  /** 是否需要用户手动配置轴/父子关系等额外语义（矩阵/树/曼陀罗等需要，星团/网不需要） */
  requiresManualSetup: boolean;

  /** 根据当前念头集合与已存坐标，计算布局位置 */
  computeLayout(
    thoughts: Thought[],
    edges: Edge[],
    existingCoords: Map<ThoughtId, GeometryCoordinate>
  ): LayoutResult;

  /** 对尚未设置坐标的念头做弱推断（半透明预览用），返回的坐标标记 isInferred=true */
  inferInitialCoordinates(
    thoughts: Thought[],
    edges: Edge[]
  ): Map<ThoughtId, GeometryCoordinate>;

  /** 判断当前一组念头是否适合结晶为该几何类型的固定形态 */
  evaluateCrystallization(structure: Structure): { canCrystallize: boolean; cohesionScore: number };

  /** 返回结晶后的可视化形态描述（供渲染层选择对应的3D/2D美术资源） */
  getCrystallizedFormAsset(structure: Structure): CrystallizedFormAsset;
}
```

每种几何结构在 `geometry-engines/{type}/index.ts` 中实现该接口，例如：

```typescript
// packages/geometry-engines/tree/index.ts
export const TreeEngine: IGeometryEngine = {
  type: 'tree',
  requiresManualSetup: true,

  computeLayout(thoughts, edges, existingCoords) {
    // 用 d3-hierarchy 的 tidy tree 算法，基于 existingCoords 中的 parentId 构建层级
    // 若某念头缺少 parentId，暂时挂在虚拟根节点下，等待用户手动归位
    ...
  },

  inferInitialCoordinates(thoughts, edges) {
    // 基于 subordinate 类型的边，推断出初始的父子关系
    ...
  },

  evaluateCrystallization(structure) {
    // 树的结晶判定：层级清晰（无孤立节点）+ 深度 >= 2 + 每个非叶节点子节点数 >= 2
    ...
  },

  getCrystallizedFormAsset(structure) {
    return { assetId: 'coral-growth', variantSeed: hashStructureId(structure.id) };
  }
};
```

**RecommendationEngine 调用方式**：遍历所有已注册的 GeometryEngine，对当前念头/边的分布跑一遍启发式规则（对应Part2第五节的表格），返回推荐分数最高的几个几何类型，UI层据此展示气泡提示。

## 六、状态管理与响应式数据流

```
用户在画布上拖拽一个念头
   │
   ▼
Interaction Layer 识别为 MoveThoughtIntent { thoughtId, newPosition, geometryType }
   │
   ▼
dispatch 到 ThoughtService.updatePosition(intent)
   │
   ▼
ThoughtService 写入 Yjs: lensCoordinatesMap.set(key, {...coord, x, y})
   │
   ▼
Yjs 触发 observe 回调（响应式）
   │
   ▼
Zustand store 中对应的 thoughtPositions 状态更新
   │
   ▼
React组件订阅该状态变化 → 触发 Pixi层重新绘制该节点位置（增量更新，不重绘整个画布）
```

**关键工程决策**：渲染层（Pixi）不直接订阅Yjs，中间必须经过Zustand这一层做"差量提取"，否则Yjs的细粒度变更事件会导致渲染层频繁触发不必要的全量刷新。Zustand store 负责把Yjs的变更聚合成"本帧需要更新哪些节点"的最小集合，交给Pixi做局部重绘。

## 七、渲染引擎详细实现方案（Web/桌面，PixiJS）

### 7.1 场景图组织

```
PIXI.Application
└── Viewport (pixi-viewport 库，处理缩放/平移手势)
    ├── BackgroundLayer      # 根据当前geometryType绘制对应网格背景
    ├── EdgeLayer            # 所有连线，按relation_type分style渲染
    ├── ThoughtLayer         # 所有念头节点（Sprite + 文字），按mass/temperature动态调整大小/颜色
    ├── StructureBoundaryLayer  # 结构容器的包围框/折叠态大节点
    └── OverlayLayer         # 拖拽中的临时预览、吸附提示线、呼吸光晕（观察模式）
```

### 7.2 性能关键实现

- **视口裁剪**：只在 `ThoughtLayer` 中渲染当前 viewport 可见范围内（含缓冲区）的节点，其余节点从场景图中移除但保留数据状态（用 `pixi-viewport` 的 `culling` 插件或手动实现四叉树空间索引 `rbush`）
- **LOD分级渲染**：缩放层级低于阈值时，用低模图标（甚至合并显示为一个"聚合圆点+数字"）代替完整节点渲染，参考地图应用的聚合点（marker clustering）算法
- **物理模拟节流**：`d3-force` 的迭代计算放在 `requestIdleCallback` 或 Web Worker 中做，避免阻塞主线程拖拽响应

### 7.3 移动端注意事项（react-native-skia）

- 手势识别需要用 `react-native-gesture-handler` 单独实现缩放/拖拽/长按，Web端的鼠标交互模型不能直接照搬（移动端拖拽念头需要先长按进入"可拖拽态"，避免和画布平移手势冲突）
- 移动端建议默认隐藏部分复杂几何结构入口（如网状/分形嵌套），聚焦"快速记录 + 简单星团/树查看"的轻量场景，复杂整理留给桌面端

## 八、AI/推荐服务实现方案

```
┌───────────────────────────────┐
│ RecommendationEngine（主线程规则判定，轻量，同步执行）        │
│  - 边类型占比统计 → 触发树/时间线/光谱等推荐                  │
└───────────────┬───────────────┘
                │ 需要语义相似度时委托
                ▼
┌───────────────────────────────┐
│ EmbeddingWorker（独立线程/Worker，异步）                    │
│  - transformers.js 加载轻量 sentence-embedding 模型         │
│  - 输入：念头文本 → 输出：向量，写入 embeddings-cache         │
│  - 用于：螺旋结构的主题相似度判定 / "意外重逢"推荐            │
└───────────────────────────────┘
```

- 首次使用相似度类功能时，需下载轻量模型（约20-80MB级别），做好"首次启用需下载模型"的用户提示
- 云端API作为可选增强渠道（用户主动在设置中开启，并明确告知数据会发送到第三方），默认关闭，保护笔记隐私

## 九、同步与多端一致性实现（Phase 4 可选）

```
本地 Yjs Doc ⇄ y-indexeddb / y-sqlite（本地持久化，Phase 1即需要）
       │
       │ Phase 4 引入：
       ▼
本地 Yjs Doc ⇄ WebSocket ⇄ y-websocket Server ⇄ 其他设备的 Yjs Doc
```

- 同步服务器可以用最简单的 `y-websocket` 官方实现自建，或用托管服务（如 Liveblocks），避免自己维护复杂的服务端状态
- 冲突解决完全交给 CRDT 语义处理（字段级自动合并），不需要额外写冲突解决逻辑——这是选择 Yjs 而非自建同步协议的核心价值
- 端到端加密：Yjs 文档在离开设备前先加密（如用 libsodium），同步服务器只存储密文，无法读取用户笔记内容

## 十二、技术风险清单与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 念头数量达到几千级后画布卡顿 | 核心体验受损 | 提前规划LOD分级渲染与视口裁剪，Phase 0就要验证这套机制，不要等到Phase 2数据量上来才补 |
| Yjs文档过大导致启动加载慢 | 影响每次打开App的体验 | 定期做Yjs文档的"压缩快照"（squash历史操作日志），避免undo历史无限增长 |
| 12种几何算法维护成本高 | 后期迭代变慢、bug面广 | 严格执行统一 `IGeometryEngine` 接口约束，配合共享的测试用例模板，保证新增/修改结构时行为一致可测 |
| 本地embedding模型体积影响首次启动体验 | 用户流失 | 该功能设计为可选延迟加载，不阻塞核心记录/整理功能的首次使用 |
| React Native画布渲染与Web表现不一致 | 多端体验割裂 | 尽早（Phase 0阶段）就搭建移动端最小原型验证 react-native-skia 的渲染能力上限，而不是留到Phase 4才发现坑 |
| CRDT同步的字段级合并在"坐标类"数据上可能产生视觉跳变（如两端同时拖拽同一节点） | 协作体验诡异 | Phase 4上线协作前，需专门设计"拖拽中的节点临时锁定"提示机制，减少并发编辑冲突的观感突兀 |

## 十三、最小可验证起点（给你一个具体的第一步）

如果要立刻动手，建议第一周只做一件事，验证整个架构假设是否成立：

**用 `core-domain` + `geometry-engines/cluster` + `render-web`（PixiJS）+ Yjs + IndexedDB，做一个能创建念头、拖拽、连线、力导向自动布局、并且刷新页面后数据还在的最小原型。**

这个原型能验证三件事：
1. Yjs作为响应式数据源驱动Pixi渲染的技术路径是否顺畅（这是整个架构的地基）
2. 力导向布局的拖拽手感是否"顺手"（Part1 Phase 0的核心判断标准）
3. Monorepo的分层约束（core-domain不依赖渲染库）是否在实际编码中可持续遵守

这三件事任何一个卡住，都值得在扩大范围前先解决，而不是带着隐患进入Phase 1。
