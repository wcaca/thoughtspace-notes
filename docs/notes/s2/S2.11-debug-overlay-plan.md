# S2.11 DebugOverlay — 把 RenderPipeline stats 可视化到屏幕

> 状态: 已实做 (2026-07-13)
> 配套: `src/v2/debug/debug-overlay.js` + `tests/v2/debug-overlay.test.js`

## WHY (为什么)

S2.10 render-pipeline 暴露了 `getStats()`，但每次排查都要在 console 跑 `__v2.pipelineStats()`。**不够直接**——AI 和开发者在浏览器里看不到实时数字，调 5 阶段参数（priority、stage 顺序）只能凭感觉。

**目标**: 持续可视化 FPS / ms / 5 阶段耗时 / cache hit rate / 错误 / 帧数，让瓶颈一眼可见。

## 约束 (来自 design review)

1. 渲染开销 < 0.5ms / 帧 (不抢主帧)
2. 默认隐藏, 不污染正常画面
3. 切换显隐时数据连续 (不跳 0)
4. 错误 / 越界有颜色高亮, 不抢戏

## 设计 (DOM-over-canvas)

| 选项 | 优势 | 劣势 |
|---|---|---|
| **DOM div + table** (选) | < 0.5ms 天然, 字体可读, 浏览器 devtools 直接 inspect | 排版受 CSS 影响 |
| Canvas 文本 | 完全自定义, 排版自由 | 5Hz 采样下省不了 0.3ms, 多一层渲染 |
| WebGL overlay | 高性能 | 完全过设计, 学习曲线 |

选 DOM 理由: **调试场景排版天然 > 极致性能**。

## 接口

```js
import { DebugOverlay } from './debug/debug-overlay.js';

const overlay = new DebugOverlay(renderPipeline, { visible: false });
overlay.attach();   // 启动 RAF 更新
overlay.toggle();   // 切显隐
overlay.detach();   // 停掉
overlay.getLastStats();  // 拿最近 stats
```

## 关键决策 (沉淀到 @note)

1. **5Hz 采样** (`refreshMs=200`): 太密看不出变化, 太稀看不出抖动
2. **console fallback**: 无 DOM 环境 (vitest node) 时降级 console.log, 避免环境分叉
3. **DOM 文本节点更新, 不重排**: _refreshDom 只 setText, 不重建 panel
4. **三层颜色编码**:
   - `v2-debug-alarm` (红) = 超 budget 或有错误
   - `v2-debug-warn` (琥珀) = 接近 budget (>80%) 或有越界
   - 默认 (白) = 正常

## 触发方式

- 顶部 [⏱] 按钮 (固定位置, 不抢主帧)
- 全局快捷键 `~` (反引号, 不会跟 input/textarea 冲突)
- 程序 API: `setVisible(true)` / `toggle()`

## 集成 (待做)

## 性能验证 (S2.12 之前)

- S2.12 expected-calculator 会算 overlay 自己的渲染成本
- 经验估算: 文本节点更新 ~0.05ms × 16 个 cell = 0.8ms (待测量)
- 目标: 实际 < 0.5ms, 不达预期则降到 2Hz
