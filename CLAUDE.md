# 念头空间 - 笔记路线 (thoughtspace-notes)

> L1 文档 - 项目宪法。所有工作开始前必读。

## 产品灵魂
用户不是在"使用"一个工具管理任务,而是在"照料"一片属于自己的念头田野。
记录是播种,整理是耕作,观察是静观其生长与凋零,行动是收获。

## 路线定位
本仓库是**笔记路线**:以笔记/念头记录与空间化整理为核心生产力工具,
交互上**模拟游戏感**(粒子/光晕/连接吸附/温度衰减/结晶动画等),
但不刻意做成"游戏产品"——游戏机制服务于内观体验,而非反过来。

> 平行仓库 `wcaca/thoughtspace-arcade` 是**游戏路线**,
> 把同一份念头数据以游戏机制(成就/探索/任务)包装成独立游戏。
> 两条路线的**领域模型与几何引擎共用同一套设计语言**(见 Part 2 几何结构),
> 但呈现层与玩法目标不同。

## 技术栈(Phase 0 起步,逐步收敛)
- 纯前端单页应用: HTML + ES Module + PixiJS + d3-force + Yjs + y-indexeddb
- 不引入框架(React/Vue)的 Phase 0 阶段,避免心智负担,验证核心交互手感
- 后期再切到 React + Vite + Tauri(桌面)+ React Native(移动)

## 不可违反的架构约束
1. **Phase 0 单文件 ≤ 600 行**:超过即触发拆分,优先按职责拆分。
2. **核心算法与渲染解耦**:力导向布局算法必须能从 PixiJS 独立调用与测试。
3. **Yjs 文档是唯一权威数据源**,SQLite/IndexedDB 均为可重建的镜像或索引。
4. **所有自动化建议以"半透明预览"形式呈现**,不得直接修改用户数据。
5. **本地优先**:笔记内容默认不上传云端,任何网络请求都需用户主动开启。

## 核心隐喻与术语表
| 中文 | 英文 | 含义 |
|---|---|---|
| 念头 | Thought | 空间中的最小可操作单元,笔记的"折叠态" |
| 笔记 | Note | 念头的"展开态",承载完整文字/图片/链接 |

| 结构 | Structure | 一组念头及其关系的容器 |
| 结晶 | Crystallization | 结构内聚度达标后触发的固定可视化形态 |
| 温度 | Temperature | 念头的新鲜度,随时间衰减,随交互回升 |
| 质量 | Mass | 被引用/连接的频次,决定视觉大小与引力 |

## 战略决策(2026-07)
**先做软件本体,平台后适配。**
- 软件本体是"念头空间"的**唯一真相源**:形态是"在用户本机运行的桌面/Web 应用"
- 任何平台适配(TapTap 小游戏 / 移动端 / Web SaaS 等)都是**可拆卸的适配层**,不是核心
- 这意味着:不因任何平台的 API 限制而妥协核心交互(如 12 种几何结构、温度衰减、Yjs CRDT)
- 平台适配启动时机:**软件本体完成 Phase 1 MVP、用户实测反馈"手感顺手"达标之后**
